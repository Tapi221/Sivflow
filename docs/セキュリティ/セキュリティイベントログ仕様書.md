# セキュリティイベントログ仕様書

本ドキュメントでは、不正アクセス検知および証跡管理のために記録する「セキュリティイベントログ」の設計仕様を定義します。

## 概要

### 目的

1. **不正アクセスの検知**: 無効化（Revoked）された端末からのアクセスや、異常な頻度の認証エラーを早期に発見する。
2. **証跡保全 (Audit Trail)**: 「いつ・誰が・どの端末で・何をしたか」を記録し、事後調査を可能にする。
3. **ユーザーへの透明性**: ログイン履歴やセキュリティアラートをユーザー自身が確認できるようにする。

### 保存ポリシー

- **保存先**: Firestore `security_logs` コレクション（Userごとのサブコレクションではなく、ルートに配置または `audit_logs/{logId}` とするが、アクセス権管理の観点から `users/{userId}/securityLogs` を採用する）
- **保存期間**: 365日（1年後に自動削除）
- **書き込み権限**:
  - Cloud Functions (Admin SDK): 全権限
  - Client (SyncService): 自分のID配下のみ `create` 許可（`update`/`delete` 禁止）

## データモデル

### パス

`users/{userId}/securityLogs/{logId}`

### スキーマ定義

```typescript
interface SecurityLog {
  logId: string;       // UUID
  userId: string;
  deviceId: string;    // アクセス元デバイスID (不明な場合は 'unknown')
  deviceName?: string; // 記録時点でのデバイス名
  
  eventType: SecurityEventType;
  severity: 'info' | 'warning' | 'critical';
  source: 'client' | 'server'; // ログの信頼度区分 (client=参考値, server=確定事実)
  isUserVisible: boolean;      // UIに表示するかどうか
  
  description: string; // ログ詳細（日本語可）
  metadata?: {
    ipAddress?: string; // (serverのみ信頼可能)
    userAgent?: string;
    path?: string;      // アクセスしようとしたリソース
    reason?: string;    // エラー詳細
    [key: string]: any;
  };
  
  occurredAt: Timestamp;
}

type SecurityEventType = 
  | 'LOGIN_SUCCESS'         // ログイン成功
  | 'LOGIN_FAILED'          // ログイン失敗
  | 'DEVICE_REVOKED'        // デバイスの登録解除（自分/他端末操作）
  | 'ACCESS_DENIED_REVOKED' // 無効化端末からのアクセス拒否
  | 'SYNC_AUTH_ERROR'       // 同期時の認証エラー
  | 'SENSITIVE_OP_REVOKED'  // 重要な操作（削除等）の失敗
  | 'DEVICE_NEW_REGISTER'   // 新規デバイス登録
  // Admin Ops
  | 'ADMIN_DEVICE_REVOKE'   // 管理者による端末解除
  | 'ADMIN_ACCOUNT_LOCK'    // 管理者によるアカウントロック
  | 'ADMIN_LOG_EXPORT';     // 監査ログのエクスポート
```

## 記録対象イベント一覧

| イベントID | EventType | Severity | 記録トリガー | Description例 | 備考 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **SEC-001** | `DEVICE_NEW_REGISTER` | info | `localStorage` に `deviceId` がなく新規生成された時 | "新規デバイスとして登録されました: Chrome (Windows)" | 初回ログイン時など |
| **SEC-002** | `LOGIN_SUCCESS` | info | Firebase Auth ログイン成功時 | "ログインに成功しました" | |
| **SEC-003** | `DEVICE_REVOKED` | **warning** | ユーザーが設定画面から端末を解除した時 | "デバイス [iPad Air] の登録を解除しました" | 解除操作を行った側の端末で記録 |
| **SEC-004** | `ACCESS_DENIED_REVOKED` | **critical** | Revoke済みの `deviceId` でSyncServiceが初期化/同期された時 | "無効化された端末からのアクセスをブロックしました" | SyncServiceV2.checkDeviceStatus() で記録 |
| **SEC-005** | `SYNC_AUTH_ERROR` | warning | Firestore セキュリティルール等で Permission Denied が返った時 | "同期権限エラー: 認証情報の不整合" | |

## 実装指針

### 1. ログ送信メソッドの拡張

既存の `TelemetryService` を拡張し、セキュリティログ専用のメソッドを追加するか、あるいは `TelemetryService` から `SecurityLogService` を分離して呼び出す。

重要なイベント（Severity: `critical`, `warning`）については、**即座に Firestore へ書き込む（バッチ処理やメモリキューを介さない）** ことを推奨する。確実に証跡を残すためである。

```typescript
// 実装イメージ
class SecurityLogger {
  async log(userId: string, event: SecurityEventType, detail: string, severity: 'info'|'warning'|'critical') {
    const logData = {
      userId,
      deviceId: getDeviceId(),
      eventType: event,
      severity,
      description: detail,
      occurredAt: serverTimestamp()
    };
    // 直接Firestoreへ書き込み（失敗してもリトライはしないが、awaitはする）
    await addDoc(collection(db, `users/${userId}/securityLogs`), logData);
  }
}
```

### 2. クライアント側の改ざん防止

クライアントから直接送信されるログは、悪意あるユーザーによって改ざんされる可能性がある（例: 他人のIDでログを送る）。
これを防ぐため、Firestore セキュリティルールで検証を行う。

```javascript
match /users/{userId}/securityLogs/{logId} {
  // 作成のみ許可（一度書いたら消せない・直せない）
  allow create: if request.auth.uid == userId
                && request.resource.data.userId == userId; // 本人のログのみ
  allow update, delete: if false; // 改ざん禁止
  allow read: if request.auth.uid == userId;
}
```

### 3. IPアドレスの記録

クライアントサイド（JS）からは正確なIPアドレスを取得できないため、クライアントから書き込むログには `ipAddress` は含めない（または `metadata` に任意で含めるが信頼しない）。
厳密なIP記録が必要なイベント（例: アカウント乗っ取り疑い）の場合は、Cloud Functions 経由での処理を検討する。

## UI表示

設定画面の「セキュリティ」タブ（将来実装）等で、このログを表示する。

- **ユーザーへの見せ方**: 専門的なログ（`ACCESS_DENIED...`）は翻訳し、分かりやすく表示する。
  - 例: ⚠️ **警告**: 2026/02/02 14:00 - 無効化された iPhone 12 からのアクセスがありました。

---

**補足**: 現状の `SyncServiceV2` の `telemetry` はメモリ内保持およびデバッグ出力が主であるため、本仕様に基づく永続化の実装は `SecurityLogService` として別途切り出すか、TelemetryService のバックエンドとして実装する必要がある。
