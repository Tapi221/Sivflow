# Phase 1.5 PoC 2 & 3 テスト・検証計画書

## 📋 検証概要

Phase 1.5で実装したPoC 2（Atomic Increment）とPoC 3（Adaptive Flush）の技術的実現可能性を検証します。

---

## 🔬 PoC 2: Atomic Increment 検証

### 検証目標
- Transaction方式と比較してスループットが5倍以上向上すること
- 仮ID→正規IDマッピング遅延が平均1秒以内であること

### テストシナリオ

#### T-PoC2-1: 基本動作確認
**目的**: Atomic Incrementの基本機能が正常に動作することを確認

**手順**:
1. Cloud Functions `getTemporarySequence` を呼び出し
2. 仮シーケンス番号が即座に返却されることを確認
3. `resolveSequence` で正規IDへのマッピング状況を確認
4. マッピング完了まで待機（最大10秒）
5. 正規グローバルSeqが取得できることを確認

**合格基準**:
- [ ] 仮シーケンス取得レスポンスタイム: 50ms以下
- [ ] マッピング完了時間: 10秒以内
- [ ] 正規グローバルSeqが一意であること

---

#### T-PoC2-2: スループット比較テスト
**目的**: Transaction方式と比較してスループットが5倍向上することを確認

**環境**:
- 同時実行ユーザー数: 10
- 各ユーザーの操作回数: 100回
- 合計操作数: 1,000回

**測定項目**:
1. **Transaction方式（Phase 1）**:
   - 総実行時間
   - 平均レスポンスタイム
   - エラー率

2. **Atomic Increment方式（PoC 2）**:
   - 総実行時間
   - 平均レスポンスタイム（仮ID取得）
   - マッピング完了率

**合格基準**:
- [ ] Atomic Increment方式の総実行時間がTransaction方式の1/5以下
- [ ] エラー率: 1%以下
- [ ] マッピング完了率: 99%以上

---

#### T-PoC2-3: マッピングキュー処理テスト
**目的**: バックグラウンドマッピング処理が正常に動作することを確認

**手順**:
1. 100件の仮シーケンスを連続発行
2. `processSequenceMappingQueue` の実行を監視
3. すべてのマッピングが完了するまでの時間を測定

**合格基準**:
- [ ] 100件のマッピングが60秒以内に完了
- [ ] マッピングエラー: 0件
- [ ] 正規グローバルSeqの重複: 0件

---

#### T-PoC2-4: 競合状況でのストレステスト
**目的**: 高負荷時でもAtomic Incrementが安定動作することを確認

**環境**:
- 同時実行ユーザー数: 50
- 各ユーザーの操作回数: 100回
- 合計操作数: 5,000回

**合格基準**:
- [ ] すべての仮シーケンスが一意であること
- [ ] エラー率: 5%以下
- [ ] マッピング完了率: 95%以上

---

## 🔬 PoC 3: Adaptive Flush 検証

### 検証目標
- ネットワーク遅延に応じてバッチサイズが動的に調整されること
- ネットワーク不安定時の再送成功率が固定設定より向上すること

### テストシナリオ

#### T-PoC3-1: RTT測定機能確認
**目的**: RTT測定が正常に動作することを確認

**手順**:
1. `AdaptiveFlushService.measureRtt()` を10回実行
2. 各測定値を記録
3. 平均RTTを計算

**合格基準**:
- [ ] 測定エラー: 0回
- [ ] RTT値が妥当な範囲（10ms〜2000ms）であること

---

#### T-PoC3-2: バッチサイズ動的調整テスト
**目的**: ネットワーク状況に応じてバッチサイズが適切に調整されることを確認

**シナリオA: 高速ネットワーク（RTT < 100ms）**
- 期待されるバッチサイズ: 10件
- 期待されるフラッシュ間隔: 300ms

**シナリオB: 中速ネットワーク（RTT 100-200ms）**
- 期待されるバッチサイズ: 20件
- 期待されるフラッシュ間隔: 500ms

**シナリオC: 低速ネットワーク（RTT > 500ms）**
- 期待されるバッチサイズ: 50件
- 期待されるフラッシュ間隔: 2000ms

**合格基準**:
- [ ] 各シナリオで期待されるバッチサイズ±10%以内
- [ ] 各シナリオで期待されるフラッシュ間隔±20%以内

---

#### T-PoC3-3: ネットワーク不安定時の再送テスト
**目的**: ネットワーク不安定時の再送成功率が向上することを確認

**環境**:
- ネットワーク遅延: 500ms±200ms（ランダム変動）
- パケットロス率: 10%
- 操作数: 100回

**測定項目**:
1. **固定バッチ設定（バッチ20件、間隔500ms）**:
   - 再送回数
   - 最終成功率

2. **Adaptive Flush設定**:
   - 再送回数
   - 最終成功率
   - バッチサイズの変動範囲

**合格基準**:
- [ ] Adaptive Flushの再送回数が固定設定より20%以上削減
- [ ] 最終成功率: 95%以上

---

#### T-PoC3-4: スループット適応テスト
**目的**: サーバー処理速度に応じてバッチサイズが調整されることを確認

**シナリオ**:
1. 初期状態: 通常スループット（50 ops/sec）
2. 高負荷状態: 低スループット（10 ops/sec）
3. 回復状態: 通常スループット（50 ops/sec）

**合格基準**:
- [ ] 高負荷時にバッチサイズが縮小（30%以上削減）
- [ ] 回復時にバッチサイズが元に戻る（±10%以内）

---

## 🧪 統合テスト

### T-INT-1: PoC 2 + PoC 3 統合動作確認
**目的**: 両PoCを同時に有効化した状態で正常動作することを確認

**手順**:
1. Atomic Incrementで仮シーケンスを発行
2. Adaptive Flushで操作をキューイング
3. ネットワーク状況を変化させながら100回の操作を実行

**合格基準**:
- [ ] すべての操作が正常完了
- [ ] データ消失: 0件
- [ ] マッピング完了率: 95%以上

## 📝 テスト実行手順

### 1. 環境準備
```bash
# Cloud Functions デプロイ
cd functions
npm run build
firebase deploy --only functions:getTemporarySequence,functions:resolveSequence,functions:processSequenceMappingQueue

# クライアントビルド
cd ..
npm run build
```

### 2. テスト実行
```bash
# PoC 2 テスト
npm run test:poc2

# PoC 3 テスト
npm run test:poc3

# 統合テスト
npm run test:integration
```

### 3. ベンチマーク実行
```bash
npm run benchmark:phase1.5
```
