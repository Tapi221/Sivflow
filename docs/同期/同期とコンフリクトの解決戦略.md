# 設計判断記録：同期とコンフリクトの解決戦略 (v2.1)

## 1. 同期の哲学的課題

オフラインファーストなアプリ（Local First App）において、同期は単なる技術的なデータ転送ではありません。
ユーザーAが地下鉄で学習し（端末A）、同時にユーザーAがPCで編集した（端末B）場合、**「どちらの変更を正とするか？」** という哲学的な問いに答える必要があります。

## 2. 採用した戦略：IndexedDB as Source of Truth & Server Wins

V2.1 では、実装の単純さとデータ整合性のバランスを考慮し、以下の戦略を採用しています。

### Base Rule: Server Wins (LWW)
ほとんどのケースでは、**「サーバー（クラウド）にあるデータ、または最後にサーバーに到達したデータ」** を優先的に採用します。
*   **理由**: 複数デバイス間での不整合（スプリットブレイン）を防ぐため、最終的な裁定権をクラウドに持たせています。
*   **実装**: 同期サービスがリモートの変更を Pull した際、`DiffEngine` がローカルデータと比較し、差分がある場合はリモートの内容でローカルを更新します。

### Exception: Data Protection via Conflict Table
単純な上書きによるデータ消失を防ぐため、システムは「紛失した可能性のあるデータ」を記録します。
*   **Conflict Logging**: リモートデータでローカルデータを上書きする際、ローカル側にも変更（未同期）があった場合は、マージ前の両データを `conflicts` テーブルに保存します。
*   **目的**: ユーザーが後から「あの時消えたかもしれないデータ」を参照・復旧できるようにするためです。

---

## 3. コンフリクト解決のUX (V2.1 実装)

### 解決フロー
1.  **自動マージ**: アプリ起動時またはバックグラウンド同期中に、`server_wins` に基づいて自動的にデータが更新されます。
2.  **サイレント記録**: 競合が発生しても、ユーザーの操作を中断させるポップアップは出しません。
3.  **事後確認**: `conflicts` テーブルに記録されたデータは、開発者ツールや将来的な「競合解決画面」から確認可能です。

### 設計思想
**「ユーザーの手を止めず、かつデータは捨てない」**。
以前の「手動解決を強いる」設計から、バックグラウンドでの自動解決＋ログ記録による事後救済へとシフトしました。
これにより、同期の遅延やネットワークの不安定さによるストレスを最小限に抑えています。
