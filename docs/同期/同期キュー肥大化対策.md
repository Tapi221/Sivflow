# 同期キュー肥大化対策

## 概要

`syncQueue`（同期キュー）が肥大化するとメモリ使用量増大、処理遅延、データ同期の停滞、ユーザ体験悪化、最悪はアプリクラッシュを招きます。
**2026年2月1日の「同期キュー基盤刷新 (Rev.5 + Critical Fixes)」実装により、自動圧縮、優先度制御、堅牢なリトライ機構、および「孤児リカバリ」が導入されました。**
本書では、新基盤に基づいた「現状把握」「検出」「恒久対策」「運用手順」を詳細に解説します。

## 2026/02/01 更新：新基盤による自動対策

最新の `OperationQueueService` では、以下の機能が標準実装されており、肥大化リスクを大幅に低減しています。

### 1. 操作合成マトリクス (Operation Synthesis Matrix)

同一 `targetId` に対する保留中の操作は、新規投入時に自動的に圧縮されます。これにより、キューの長さが物理的に抑制されます。

- `Create` + `Update` = `Create` (最新データでの作成に集約)
- `Update` + `Update` = `Update` (最新データの上書き)
- `Create` + `Delete` = `Discard` (キューから消滅：サーバー通信ゼロ)
- `Delete` + `Create` = `Update` (事実上の復元)

### 2. 優先度付き多段取得 (Cascading Priority Fetch)

`High` > `Normal` > `Background` の順で処理枠を割り当てます。バックグラウンド処理が大量に溜まっても、ユーザーの直接的な操作（High）がブロックされるのを防ぎます。

### 3. 並列実行制御 (Strict Concurrency Limit)

`MAX_CONCURRENCY = 5` を厳守し、ブラウザの通信リソースを食いつぶさないように設計されています。

### 4. 孤児リカバリ (Stale Processing Recovery) 【New!】

アプリが強制終了した場合などに、`processing` 状態のまま永遠に残るアイテム（孤児）を自動検出します。

- **ロジック**: `processing` 状態で 5分 以上経過したアイテムを検出し、自動的に `pending` 状態へロールバックします。
- **効果**: 「同期が永遠に終わらない（スタックする）」現象を自己修復します。

## 現状（想定される症状）

- **DLQ（Dead Letter Queue）の蓄積**: 特定のアイテムが 5 回失敗し、`syncErrors` テーブルに移動して滞留する。
- **特定対象への集中**: 大量編集により、特定のカードやフォルダに対する操作が連続して発生する（新基盤の合成で緩和済み）。
- **Processingスタック**: クラッシュやバグにより、再起動後も処理中マークが外れない（**孤児リカバリで自動解決済み**）。

## 監視・検出（IndexedDB クエリ）

新基盤では IndexedDB の `@dexie` インデックスが最適化されているため、以下のクエリで状況を正確に把握できます。

### 1. キュー長の確認

```javascript
// 全体の保留数（pending）を確認
const pendingCount = await db.syncQueue.where('status').equals('pending').count();
// 優先度別の分布
const highPrio = await db.syncQueue.where('[status+priority]').equals(['pending', 'high']).count();
```

### 2. アイテムの鮮度確認

```javascript
const oldest = await db.syncQueue.orderBy('createdAt').first();
const ageMs = Date.now() - oldest.createdAt;
console.log(`最古アイテムの滞留時間: ${ageMs / 1000 / 60} 分`);
```

### 3. スタック（孤児）の検出

Rev.5.1 の自動リカバリが機能していない場合の手動確認用

```javascript
const stuckItems = await db.syncQueue
    .where('status').equals('processing')
    .filter(item => item.processingStartedAt && (Date.now() - item.processingStartedAt > 300000))
    .toArray();
console.log('スタック数:', stuckItems.length);
```

### 4. DLQ（失敗履歴）の確認

```javascript
const totalErrors = await db.syncErrors.where('phase').equals('queue_dlq').count();
```

## 運用手順（インシデント対応フロー）

1. **診断**: `db.syncQueue` の状態を DevTools (IndexedDB) で確認。
2. **分離**: `status = 'processing'` のまま固定されているアイテム（クラッシュによるアンロック失敗）がないか確認。
   - **Rev.5.1以降は自動修復されます** が、念のため確認。
3. **分析**: `lastError` フィールドを見て、サーバー側のエラー（5xx）かネットワークエラーか、シリアライズエラーかを判別。
4. **復旧**:
   - ネットワーク改善を待つ。
   - DLQ (`syncErrors`) に退避されたアイテムを管理画面から確認する（現時点ではログ確認のみ）。

## 恒久対策（ベストプラクティス）

1. **ペイロードの軽量化**
   - 大きなデータ（画像など）は直接 `images` テーブルで管理し、`syncQueue` には `id` のみを載せる（参照型ペイロード）。
2. **監視サービスの導入**
   - `SyncHistory` を定期的に解析し、成功率 99% 未満が続く場合にユーザーに通知する。
3. **バックプレッシャーの高度化**
   - キューが 1000 を超えた場合、UI側で編集操作を一時制限し、「データの同期を待機中...」というトーストを表示。
