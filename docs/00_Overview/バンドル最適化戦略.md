# フロントエンド設計記録：バンドル最適化と分割戦略

## 1. 課題と背景

FlashcardMasterは「学習アプリ」としての特性上、以下の相反する要件を抱えていました。

1.  **初期表示の軽快さ**:
    ユーザーは「今日の復習」をすぐに始めたい。アプリを開いてから数秒間待たされるのは致命的なUX欠陥となる。
2.  **機能のリッチさ**:
    数式（KaTeX）、コードハイライト（Prism）、グラフ描画（Recharts）、画像加工など、学習を支援する高度な機能には「重いライブラリ」が不可欠。

この矛盾を解決するため、Viteのビルド設定において**「戦略的チャンク分割」**を実施しました。

---

## 2. 分割の意思決定プロセス

### Editor (Katex, Prism) をなぜ分けたか
編集画面や学習画面でしか使われないライブラリ群（`vendor-editor`）は、**約1.3MB**にも及びます。
これをメインバンドルに含めると、ユーザーが単にダッシュボードを確認したいだけでも、裏で巨大なJSファイルのダウンロードと解析が行われてしまいます。
これを「編集ボタンを押すまで読み込まない」ように分離することで、初期ロード（Total Blocking Time）を劇的に改善しました。

### Charts (Recharts) をなぜ分けたか
統計画面は重要ですが、毎日必ず全員が見るわけではありません。
`vendor-charts`（**約333KB**）を分離することで、メインフローの軽さを優先しました。

### Firebase (Auth/Firestore) をなぜ分けなかったか
Firebase SDKは内部で多くのコードを共有しており、無理に分割するとコードの重複や初期化タイミングの複雑化を招きます。
「インフラは一括で読み込む」という判断の下、`vendor-firebase`としてまとめています。ただし、Storage（画像アップロード）については将来的に動的インポート化する余地を残しています。

---

## 3. 「完全自動化」をしなかった理由

今回採用した `vite.config.ts` の `manualChunks` 設定は、以下のような「関数型アプローチ」です。

```typescript
manualChunks: (id) => {
  if (id.includes('katex') || id.includes('prism')) return 'vendor-editor';
  // ...
}
```

ここで重要なのは、**「すべてを自動で最適化しようとしなかった」**点です。

### 意図的に残した「人間による監視」
例えば、「NPMパッケージのサイズを見て自動で分割するプラグイン」などを導入すれば、設定ファイルを書かずに分割できるかもしれません。
しかし、それは以下のリスクを孕みます。

*   **コンテキストの喪失**: 
    「なぜこのライブラリがこのチャンクに入ったのか」がブラックボックス化する。
*   **予期せぬ副作用**: 
    依存関係の順序が崩れ、実行時エラーになる可能性がある。

あえて `includes('katex')` のように明示的に記述することで、**「開発者が意図を持ってライブラリを分類する」**というプロセスを強制しています。
これは、将来チーム開発になった際や、数年後にメンテナンスする際、**「このアプリにとって何がHeavyなのか」**をコード自体が語るドキュメントとしての役割を果たします。

### 「95%の正解」を目指す
完全な最適化（100点）を目指して設定を複雑にするよりも、95点で手を打ち、残りの5%（新しいライブラリを追加した時のチェックなど）を人間の運用でカバーする設計にしました。
これが、長期的にメンテナンス可能な「枯れた技術選定」のあり方です。

---

## 4. 検証方法

ビルド後の `dist/assets` を確認し、以下のファイル構成になっていれば設計は成功しています。

- `index-xxxx.js`: コアロジック（300KB以下が望ましい）
- `vendor-react-xxxx.js`: React本体
- `vendor-editor-xxxx.js`: **これが最大サイズであるべき**
- `vendor-charts-xxxx.js`

この構成が維持されている限り、FlashcardMasterは機能追加を続けても、初期ロードの速さを損なうことはありません。
