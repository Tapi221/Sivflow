# フロントエンド設計記録：セキュリティと権限設計の思想

## 1. ゼロトラスト・クライアント (Trust No Client)

FlashcardMasterはフロントエンド主導（Serverless）のアプリケーションですが、**「クライアントサイドのコードは改ざん可能である」** という前提で設計されています。

### UIの役割と限界
JavaScriptで行っている「ログインしていなければボタンを隠す」「自分以外のフォルダは見せない」といった制御は、あくまで **UX（使いやすさ）のため** のものであり、セキュリティではありません。
技術的な知識がある攻撃者は、これらの制御をバイパスしてAPIを叩くことができます。

---

## 2. 最後の砦：Firestore Security Rules

真のセキュリティは、Firebaseサーバー側のルール（`firestore.rules`）によって担保されています。

### 基本哲学：Owner Only
私たちのデータモデルはシンプルです。**「自分のデータは自分しか読み書きできない」**。

```javascript
match /users/{userId}/{document=**} {
  allow read, write: if request.auth != null && request.auth.uid == userId;
}
```

この「パスベースの所有権確認」により、万が一クライアントコードにバグがあっても、AさんがBさんの学習データを覗き見ることは原理的に不可能です。

### 例外：公開機能への備え
将来的には「デッキの共有」などの機能が入るかもしれません。
その際は、「誰でも読める（`read: if resource.data.public == true`）」が、「書けるのは所有者だけ」というルールを追加します。
今のところは「Private by Default」を徹底しています。

## 3. インジェクション対策

### NoSQL Injection
FirestoreのSDKを使用している限り、クエリパラメータのエスケープは自動で行われます。
しかし、ユーザー入力（カードの内容など）をそのまま `dangerouslySetInnerHTML` で表示する箇所（特に数式やMarkdown）には細心の注意を払っています。
Reactの標準機能によるサニタイズに加え、リッチテキスト表示時は信頼できるライブラリのみを使用し、XSSを防いでいます。
