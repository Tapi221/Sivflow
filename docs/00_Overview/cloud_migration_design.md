# クラウド移行・同期システム設計 (Cloud Migration & Sync Design)

## 1. 目的
現行の「設定画面からの手動/自動バックアップ」方式から、**「常時接続・リアルタイムに近い同期システム」** への移行を目指します。
これにより、複数端末（PC, スマホ）でのシームレスな学習体験を実現し、データの堅牢性を向上させます。

## 2. 現状の課題 (As-Is)
*   **完全上書き**: バックアップ復元時、ローカルデータをクラウドデータで完全に上書きするため、端末間の差分が失われる。
*   **手動依存**: ユーザーが意識して同期ボタンを押す、またはタイマー設定に依存しており、同期忘れのリスクがある。
*   **コンフリクト**: 同時編集時の競合解決ロジックが存在しない。

## 3. 新アーキテクチャ (To-Be)

### 3.1 "Local-First" Sync Strategy
*   **Single Source of Truth**: 各クライアントの `LocalDB` (IndexedDB) を正とし、クラウドは「中継・永続化ハブ」として機能させる。
*   **Push-Pull Model**:
    *   **Push**: ローカルでの変更（カード追加、学習記録）を即座に（またはバッチで）クラウドへ送信。
    *   **Pull**: 定期的に、または特定イベント（アプリ起動、Push成功後）にクラウドから差分を取得。

### 3.2 データ構造の変更
同期を効率化するため、各エンティティに以下のメタデータを追加・活用します。

*   `updatedAt`: 最終更新日時（ミリ秒精度）。
*   `deleted`: 論理削除フラグ（物理削除は行わず、同期完了後にクリーンアップ）。
*   `version`: 楽観的ロック用のバージョン番号（コンフリクト検知用）。

### 3.3 API / サービス構成
既存の `SyncService` を拡張、または `AdvancedSyncService` として再実装します。

1.  **Operation Queue (Queue System)**
    *   オフライン時の変更操作（Create, Update, Delete）を `SyncQueue` テーブルに蓄積。
    *   オンライン復帰時に順次実行。

2.  **Conflict Resolution (競合解決)**
    *   **Last Write Wins (LWW)**: 基本戦略。`updatedAt` が新しい方を採用。
    *   **Merges**: 学習履歴などは配列結合でマージ可能な場合がある。
    *   **Manual**: 解決不可能な競合（同じカードのテキストを同時変更など）は、ユーザーに選択させるUI (`ConflictDialog`) を提供。

## 4. 移行ステップ (Migration Plan)

### Phase 1: データ構造の準備
1.  既存データに対し、`updatedAt` の整合性チェックを行うマイグレーションスクリプトを実行。
2.  論理削除 (`deleted` フラグ) のサポートを UI/Backend に追加。

### Phase 2: Sync Service の拡張
1.  `SyncQueue` の実装。
2.  差分アップロード (Delta Upload) の実装。今の「全データJSON化」をやめ、変更があったレコードのみを Firestore/Storage に送る。

### Phase 3: 競合解決UIの実装
1.  競合発生時にユーザーへ通知する仕組み。
2.  競合データを比較・選択する画面の実装。

## 5. リスクと対策
*   **データ消失**: 移行期は「旧バックアップシステム」と「新シンク」を併用し、いつでも旧方式で復元できるようにする（Safety Net）。
*   **無限ループ**: 同期トリガーが相互に呼び合わないよう、同期起因の更新は同期トリガーを引かないフラグ管理を徹底する。
