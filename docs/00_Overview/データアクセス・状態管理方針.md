# フロントエンド技術設計：データアクセスと状態管理

## 1. Local-First Architecture

FlashcardMasterは **「オフラインファースト」** を設計の主軸としています。
サーバー（Firebase）はあくまでバックアップと同期の手段であり、**真実のソース（Source of Truth）はクライアント端末にある IndexedDB です**。

### なぜ Local First か？
1.  **爆速UX**: ネットワーク待ち時間ゼロでカードが表示される。
2.  **可用性**: 地下鉄でも機内でも学習ができる（学習アプリとしての必須要件）。

### 技術構成
*   **Dexie.js (IndexedDB wrapper)**: クライアントデータベース。数万枚のカードも一瞬で検索可能。
*   **Firebase Firestore**: クラウドデータベース。デバイス間同期用。

---

## 2. 状態管理の戦略層 (State Strategy Layers)

アプリ内の状態管理は、データの性質によって以下の3層に分けて管理しています。

### Layer 1: Persistent Domain Data (カード・フォルダ)
**利用ツール**: `Dexie.js` + `useLiveQuery`
*   最も重要なデータ。
*   Reactの状態管理（useState）には極力乗せず、DBからのライブクエリ結果をそのまま描画します。
*   これにより、どのコンポーネントで更新が行われても、DBさえ更新すれば画面全体が整合性を保ちます。

### Layer 2: Server State (ユーザー情報・統計ログ)
**利用ツール**: `TanStack Query (React Query)`
*   `useQuery` を使用してキャッシュ・再取得を管理します。
*   Auth情報や、同期頻度の低い重いデータ（Global Statistics）に適用されます。

### Layer 3: UI State (モーダル開閉・入力フォーム)
**利用ツール**: `useState`, `Context API`
*   URLパラメータ（`useSearchParams`）も積極的に利用します。
    *   例: フォルダを開く際、`/FolderView?id=xxx` とすることで、リロードしても状態が維持されます。

---

## 3. 主要Hooksの責務

### `useCards()`
このフックは単なるデータ取得以上の責務を持っています。
*   **隠蔽**: コンポーネントに対し、データがDexieから来ているのかFirestoreから来ているのかを隠します。
*   **整合性**: 削除時に関連する学習ログや画像も消すといった「カスケード処理」の実装場所です。

### `useSync()` (同期システム)
*   バックグラウンドで静かに動作します。
*   「ユーザーが保存ボタンを押した時」ではなく、「適当なタイミング」で差分をサーバーに送ります。
*   コンフリクト発生時は、`lastUpdatedAt` を比較し、常に**「新しい方」**を勝たせるシンプルなLWW (Last Write Wins) 戦略を採用しています（必要に応じて手動解決UIを表示）。
