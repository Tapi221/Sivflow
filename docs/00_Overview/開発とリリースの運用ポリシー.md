# 開発・リリース運用ポリシー

## 1. 開発フロー (Development Flow)

個人開発（または少人数チーム）としての機動力を維持しつつ、品質を担保するための軽量なGit運用を採用します。

### ブランチ戦略
*   `main`: **Production Ready**。常にデプロイ可能な状態。
*   `develop`: **Staging**. 次期リリースのための統合ブランチ。
*   `feature/xxx`: 新機能開発。
*   `hotfix/xxx`: `main` で見つかった緊急バグ修正。

### Definition of Done (完了の定義)
機能開発が完了したと言える基準：
1.  ローカルで正常に動作すること。
2.  `npm run build` がエラーなく通ること。
3.  関連するドキュメント（仕様書、API定義）が更新されていること。
4.  **「美しいか？」** (Aesthetics Check): デザインがガイドラインに沿っているか。

---

## 2. リリースチェックリスト (Release Checklist)

デプロイ前に「人間が目視で確認すべき」最低限のリストです。

### 基本動作
- [ ] ログイン・ログアウトができるか。
- [ ] カードの作成・編集・削除ができるか。
- [ ] 画像のアップロードができるか（特にモバイル）。

### パフォーマンス & バンドル
- [ ] `npm run build` の出力を見て、`vendor-editor` 等の巨大チャンクが意図せずメインバンドルに混入していないか。
- [ ] Networkタブで、初期ロード時に不要なリクエストが飛んでいないか。

### エッジケース
- [ ] オフライン状態でアプリを開き、閲覧できるか。
- [ ] 存在しないページにアクセスした時、404（NotFound）またはホームへのリダイレクトが機能するか。

---

## 3. トラブルシューティング哲学

バグが発生した場合の行動指針：

1.  **止血する (Stop Bleeding)**:
    ユーザーデータが消失・汚染される恐れがある場合は、機能を停止するか、メンテナンスモードに切り替える。
2.  **再現させる (Reproduce)**:
    「なんとなく直った」は禁止。再現手順を確立してから修正する。
3.  **仕組みで防ぐ (Prevent)**:
    個人のミスで片付けず、「Lintで落とせないか？」「型定義で防げないか？」を考える。

---

## 4. バージョニング

SimVer (Semantic Versioning) に準拠します。
*   `v1.0.0`: メジャーリリース。UIの大幅刷新や互換性のないデータ構造変更。
*   `v1.1.0`: 新機能追加（エクスポート機能など）。
*   `v1.1.1`: バグ修正。

タグ付け (`git tag v1.1.0`) を推奨します。
