# プロジェクト構成とディレクトリの責務

## 1. ディレクトリ構造の哲学 (Structure Philosophy)

FlashcardMasterは、機能の拡大に伴い「機能単位（Feature-based）」のコンポーネント管理を採用しています。
`src/Components/ui` のような汎用部品と、`src/Components/card` のようなドメイン固有部品を明確に分離することで、依存関係のスパゲッティ化を防いでいます。

## 2. ディレクトリ詳細定義

```text
src/
├── Components/          # Reactコンポーネント群
│   ├── ui/             # 【Atomic】ドメイン知識を持たない汎用UI (Radix UI, Shadcn等)
│   ├── layout/         # 【Template】ヘッダー、サイドバーなどの枠組み
│   ├── card/           # 【Feature】カード編集・表示に特化した部品 (Editor, CodeBlock等)
│   ├── stats/          # 【Feature】統計・グラフ関連の部品 (Charts, Heatmap)
│   ├── export/         # 【Feature】データエクスポート・インポート機能
│   ├── math/           # 【Feature】数式レンダリング (KaTeX)
│   └── review/         # 【Feature】学習モード固有のUI (Flashcard, Controls)
│
├── Pages/              # ルーティングに対応する最上位コンポーネント (Dashboard, Study, etc.)
│
├── contexts/           # グローバル状態・DI (Auth, Dialogs)
│
├── hooks/              # カスタムフック
│   ├── useCards.ts     # カードデータのCRUDとキャッシュ戦略を隠蔽
│   └── useFolders.ts   # フォルダ階層ロジックの隠蔽
│
├── services/           # 外部/内部サービスとの境界
│   ├── firebase.ts     # Firebase初期化
│   ├── localDB.ts      # Dexie.js (IndexedDB) 定義
│   └── reviewAlgorithm.ts # 復習計算ロジック (Pure Function)
```

## 3. ガードレール (Rules)

### UI vs Feature
*   `Components/ui` 内のファイルは、`firebase` や `localDB` を import してはいけません。これらは純粋な見た目のみを担当すべきです。
*   データフェッチやビジネスロジックは `hooks` または `Pages`、あるいは `Feature` コンポーネントに持たせてください。

### Pages vs Components
*   `Pages` はコンポーネントの配置（Layout）とデータの受け渡し（Wiring）に専念し、複雑なUIロジックは `Components` に切り出してください。

### Service Isolation
*   コンポーネントが直接 `IndexedDB` を操作することは可能な限り避け、`useCards` などのフックを経由してください。これにより、将来的にバックエンドが変更された場合の影響範囲を最小化します。
