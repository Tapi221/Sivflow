# Git リリース / タグ運用ルール

このドキュメントは、本プロジェクトにおける Git ブランチ・タグの運用ルール を明確にし、
将来の自分が迷わず リリース・ロールバック できる状態を保つためのものです。

本プロジェクトは 個人開発・Firebase Hosting 前提 とし、
過剰なチーム開発向けルールは採用しません。

---

## 1. ブランチ運用ルール

### 1.1 本番ブランチ

- 本番相当ブランチは main に統一する
- Firebase Hosting に deploy するコードは、必ず main から行う
- production branch = main

### 1.2 その他のブランチ

- 作業用・検証用ブランチは一時的に作成してよい
- 本番リリース後、不要になったブランチは削除してよい
- 長期間残す前提のブランチは作らない

---

## 2. タグ運用ルール（最重要）

### 2.1 本番リリースタグ（正式）

本番リリースを示すタグは 日付ベースで固定する。

タグ形式:
  prod-YYYY-MM-DD

例:
  prod-2026-02-09
  prod-2026-02-15

採用理由:
- 時系列が一目で分かる
- バージョン番号（v1.2.3 等）の混乱を避けられる
- ロールバック対象を即座に特定できる
- 個人開発・即デプロイ運用に最適

---

### 2.2 Release Candidate（任意）

本番直前の確認用タグを付けたい場合のみ使用する。

タグ形式:
  prod-YYYY-MM-DD-rc.N

例:
  prod-2026-02-09-rc.1
  prod-2026-02-09-rc.2

- rc は 本番未満 を意味する
- Firebase 本番データを変更する deploy には使用しない
- あくまで最終確認・差分チェック用

---

### 2.3 開発用タグ（任意・削除前提）

一時的な検証・メモ用途のみ。

タグ形式:
  dev-YYYY-MM-DD

- 長期保存しない
- 履歴管理の主役にはしない

---

## 3. 採用しない運用（明示的に禁止）

以下は 本プロジェクトでは採用しない。

- セマンティックバージョニング（v1.2.3）
- hotfix / patch / minor / major の分類
- タグ名に設計思想や変更内容を詰め込むこと
- チーム開発前提の複雑なリリースフロー

理由:
- 規模に対して複雑すぎる
- 運用コストが上がり、判断が遅れる
- 個人開発ではメリットがない

---

## 4. 実際の運用コマンド

### 4.1 本番リリース時

実行手順:
  git checkout main
  git pull origin main
  git tag -a prod-2026-02-09 -m "Production release 2026-02-09"
  git push origin prod-2026-02-09

### 4.2 タグ一覧確認

  git tag --list --sort=-creatordate

### 4.3 ロールバック

  git checkout prod-2026-02-09

必要であれば、その状態から新しい修正ブランチを切る。

---

## 5. 現在の前提状態（2026-02 時点）

- 本番ブランチ: main
- 旧タグ（v2.x 系）は履歴として残す
- 今後の正規リリースタグは すべて prod-YYYY-MM-DD

---

## 6. このルールの目的

- 考えなくても正しく deploy できる状態を作る
- 未来の自分が Git 履歴を見て即理解できるようにする
- Git を作業の邪魔ではなく安全装置として使う

このルールに例外は作らない。
