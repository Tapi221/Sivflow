# デバイス管理・基本設計仕様書

本ドキュメントでは、複数端末での同期制御およびセキュリティ管理のための「デバイス認識・管理」の仕様について定義する。

## 1. 「デバイス」の定義と認識メカニズム

### 1-1. 論理デバイスとしての定義

本システムにおける「デバイス」とは、物理的なハードウェア（筐体）ではなく、**ブラウザまたは PWA アプリのインストール単位（論理デバイス）**を指す。

* **識別子**: `deviceId` (UUID v4 等で生成)
* **保存場所**: ブラウザの `localStorage`

> [!WARNING]
> **永続性に関する制約**
> 以下の場合、物理的に同じ端末であっても「**新規デバイス**」として扱われる（`deviceId` が再生成される）。
> * Safari プライベートブラウズ / Chrome シークレットモードでのアクセス
> * ユーザーによるブラウザの「履歴・Cookie・データ削除」操作
> * iOS/iPadOS のストレージ自動整理による PWA データ削除
> * ブラウザのアンインストールと再インストール

### 1-2. 識別情報構造

論理と表示を厳密に分離する。

* **Logic (制御用)**: `deviceId`
  * システム内部での識別、認証、Sync 制御にのみ使用。
  * ユーザーには不可視。不変（削除されるまで）。
* **Display (表示用)**: `deviceName`
  * UI 表示専用。
  * ユーザーによるリネームが可能。
  * 初期値は UserAgent から自動生成されるが、あくまで「初期ラベル」の扱い。

## 2. デバイス情報の自動識別と表示

### 2-1. 自動識別ポリシー（"Fuzzy Identification"）

UserAgent 情報は偽装や類似が多いため、**断定的な表現を避け、カテゴリレベルの表示**に留める。誤った詳細情報はユーザーの混乱と不信を招くためである。

| 識別ソース | 推奨表示文言（例） | NG例（断定しすぎ） |
| :--- | :--- | :--- |
| **PWA (Standalone)** | `App (Pixel 7)`, `App (iPhone)` | `Chrome`, `Safari` |
| **iPadOS Safari** | `Safari (iOS系)` | `Safari (iPad)` (Macとして判定されるため) |
| **Desktop Chrome** | `Chrome (Desktop)` | `Chrome (Windows 11)` |
| **Mobile Safari** | `Safari (iPhone)` | - |

* **アイコン表示**: 識別結果に基づき、`SmartPhone`, `Tablet`, `PC`, `App` 等の汎用アイコンを表示する。

### 2-2. デバイスリストのソート順

ユーザーの利便性を最優先し、**最終利用日時（`lastUsedAt`）の降順**で表示する。

1. **現在利用中のデバイス** (最上位)
2. 最近アクセスしたデバイス
3. ...
4. 長期間未アクセスのデバイス

#### 更新タイミングについて

`lastUsedAt` は頻繁な更新によるデータベース負荷（Write）を避けるため、以下のタイミングでのみ更新する。

* **認証トークンのリフレッシュ成功時**
* **同期（Sync）処理の完了時**
* ※ 単なる画面遷移やキーボード入力などのクライアント操作では更新しない。

### 2-3. 「現在使用中」の明示

誤削除防止のため、現在アクセスしているセッションに対応するデバイスには視覚的なインジケータを付与する。

* バッジ表示: **「現在使用中」** (Current Device)
* スタイル強調: 行背景色の変更やボーダー強調など。

## 3. デバイス管理アクション

### 3-1. デバイス名のリネーム

自動生成された名前が直感的でない場合（例: `Chrome (Windows)` が複数ある場合など）、ユーザーは任意の名前に変更できる。

* **対象**: `deviceName` フィールドのみ更新。
* **影響範囲**: 全デバイスのリスト表示に即座に同期・反映される。

### 3-2. デバイスの登録解除（ログアウト）

#### A. アクションフロー

1. ユーザーがデバイス一覧から「×」または「ゴミ箱」アイコンを押下。
2. **確認ダイアログ**を必ず表示する。
   > 「このデバイス（[デバイス名]）の登録を解除しますか？ これにより、そのデバイスでのログイン状態が無効になります。」

#### B. 解除時の挙動（Context-Aware）

| 対象デバイス | 挙動仕様 |
| :--- | :--- |
| **他デバイス** | 1. サーバー上の該当 `deviceId` レコードを**論理削除（無効化）**する。<br>   * `status: active` -> `status: revoked`<br>   * `revokedAt` タイムスタンプを記録。<br>2. 該当デバイスからの次回アクセス時に認証が拒否され、強制ログアウトされる。 |
| **現在使用中のデバイス**<br>（自分自身） | 1. **即時ログアウト処理**を実行。<br>2. サーバー上のレコードを**論理削除**（`revoked`）。<br>3. クライアント側の `deviceId` および認証トークンを破棄。<br>4. ログイン画面へリダイレクト。<br>5. **再ログイン時は「新規デバイス」として登録される。** |

#### C. 解除済み（Revoked）デバイスの表示
*   **リスト表示**: 解除後も「履歴」としてリストには残す。
*   **操作制限**: グレーアウト表示等を行い、**一切の操作（リネーム・再有効化など）を不可**とする。「復活」はさせない。

> [!TIP]
> **なぜ論理削除か？**
> デバイスレコードを物理削除せず `revoked` ステータスで残すことで、以下のメリットがある。
> * 不正アクセスの証跡として残る。
> * 無効化したはずの `deviceId` が再利用された場合の異常検知が可能になる。
>   * `revoked` 状態の `deviceId` でアクセスがあった場合、**セキュリティイベントとして記録し、ログインは拒否する**。
> * 「勝手に消えた」等の問い合わせ調査時にログとして活用できる。

## 4. 将来的な拡張（Phase 2以降）

* **デバイス数上限**: アカウントプランに応じた同時利用デバイス数の制限（例: Free版は2台まで）。
* **不正利用検知**: 短時間に多数の異なる `deviceId` からのアクセスがあった場合のアラート。
* **自動クリーンアップ**: 最終利用から一定期間（例: 90日）経過したデバイスの自動削除。
