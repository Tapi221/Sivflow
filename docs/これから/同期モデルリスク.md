# Firestore + IndexedDB 同期モデル：潜在的リスクと運用上の注意点

## 1. 潜在的リスク

1. **データ不整合リスク**
   - SyncService の差分同期ロジックにバグがあると、ローカルとクラウドでデータが食い違う。
   - オフライン中の編集を複数端末で行った場合、競合解決が失敗する可能性。

2. **Firebase依存リスク**
   - 全データをFirestoreに依存しているため、サービス障害時は全ユーザーに影響。
   - 外部クラウドとのハイブリッド利用が消えたことで、ユーザーが直接データを操作できない。

3. **コスト増加リスク**
   - Firestore/Blob Storage の課金はデータ量・リクエスト数に比例。
   - カード数や画像数が増えると、想定以上の運用コストが発生する可能性。

4. **バックアップ・復元リスク**
   - エクスポート機能は OneDrive/GDrive に限定。
   - 標準ユーザー操作では、フルバックアップや復元手順が直感的でない可能性。

---

## 2. 運用上の注意点

1. **同期ロジックの徹底テスト**
   - 双方向同期の競合ケースを網羅するテストケースを用意。
   - ローカルとクラウドの整合性チェック機能を運用に組み込む。

2. **データバックアップの強化**
   - 定期的に自動バックアップを行う仕組みを整備。
   - バックアップ先がクラウドのみで偏らないよう、オフラインコピーやユーザー手動エクスポートも推奨。

3. **コスト監視**
   - Firestore/Blob Storage 使用量を定期監視。
   - 大量データや画像アップロード時はストレージ最適化（圧縮・古いデータ削除）を検討。

4. **ユーザー通知・障害対応**
   - 同期失敗やクラウド障害発生時にユーザーにわかりやすく通知。
   - ローカルデータ保護のため、操作ログや修復手順を用意。

5. **バージョン管理・マイグレーション計画**
   - データスキーマ変更時のマイグレーション手順を明確化。
   - 古いデータとの互換性を確保する運用フローを作成。

---

## 3. 補足

- このモデルは堅牢でUX向上には有効だが、「Firebase依存」「コスト増」「同期バグ」の三大リスクを運用でカバーする必要あり。
- 大規模リリース前には、**テスト・バックアップ・コスト監視・ユーザー通知**を最低限整備すべき。
