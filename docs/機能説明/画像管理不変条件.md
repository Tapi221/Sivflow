# 画像管理ドメインの不変条件

## 絶対に破ってはいけない真実

このドキュメントは、FlashcardMaster における画像管理の **不変条件（Invariants）** を定義します。
これらのルールは、コードよりも強い制約であり、**いかなる状況でも違反してはなりません**。

---

## 不変条件（Invariants）

### 1. DBに保存されるデータに Base64文字列は存在してはならない

**理由:**
- Base64 は約1.33倍に肥大化する
- DB のサイズ制限に抵触する
- 同期速度が著しく低下する
- 差分同期が意味をなさなくなる

**適用範囲:**
- Firestore のすべてのドキュメント
- IndexedDB (Dexie) のすべてのテーブル
- 同期キュー内のデータ

**違反例:**
```ts
// ❌ NG
const card = {
  questionImages: [{
    remoteUrl: 'data:image/png;base64,iVBORw0KG...'  // Base64
  }]
};
```

**正しい例:**
```ts
// ✅ OK
const card = {
  questionImages: [{
    remoteUrl: 'https://firebasestorage.googleapis.com/...'  // Storage URL
  }]
};
```

---

### 2. `remoteUrl` は Storage由来の https URL のみ

**理由:**
- `remoteUrl` は「クラウドに存在する画像」を意味する
- Base64 や Blob URL は「リモート」ではない
- 型安全性を保つため

**検証方法:**
```ts
const isValidRemoteUrl = (url: string) => 
  url.startsWith('https://') && 
  url.includes('firebasestorage.googleapis.com');
```

**違反例:**
```ts
// ❌ NG
remoteUrl: 'data:image/png;base64,...'
remoteUrl: 'blob:http://localhost:5173/...'
remoteUrl: 'file:///C:/Users/...'
```

**正しい例:**
```ts
// ✅ OK
remoteUrl: 'https://firebasestorage.googleapis.com/v0/b/...'
```

---

### 3. `localUrl` は Blob URL のみ（data: 禁止）

**理由:**
- `localUrl` は一時的なプレビュー用
- Blob URL はメモリ効率が良い
- Base64 は肥大化するため不適切

**検証方法:**
```ts
const isValidLocalUrl = (url: string) => 
  url.startsWith('blob:');
```

**違反例:**
```ts
// ❌ NG
localUrl: 'data:image/png;base64,...'
```

**正しい例:**
```ts
// ✅ OK
localUrl: 'blob:http://localhost:5173/abc-123-def'
```

---

### 4. 画像は Storageに存在して初めて remote と呼ばれる

**理由:**
- 「リモート」の定義を明確にする
- ローカルのみの画像は `localUrl` のみを持つ
- 同期状態を明確に区別する

**状態遷移:**
```
[新規作成]
  localUrl: blob:...
  remoteUrl: null
  status: 'uploading'

[アップロード完了]
  localUrl: blob:...  (optional)
  remoteUrl: https://...
  status: 'ready'

[同期完了・ローカル削除]
  localUrl: null
  remoteUrl: https://...
  status: 'ready'
```

---

### 5. DB更新は 画像アップロード完了後のみ許可

**理由:**
- データの一貫性を保つ
- 「画像がない状態」でカードが保存されるのを防ぐ
- ロールバックを容易にする

**正しいフロー:**
```ts
// 1. 画像をアップロード
const remoteUrl = await uploadToStorage(image);

// 2. DB を更新
await updateCard({
  questionImages: [{ remoteUrl, status: 'ready' }]
});
```

**違反例:**
```ts
// ❌ NG: DB を先に更新
await updateCard({ questionImages: [{ status: 'uploading' }] });
await uploadToStorage(image);  // 失敗したらDBが不整合
```

---

## 実行時の保証

これらの不変条件は、以下の方法で保証されます:

1. **型システム（コンパイル時）**
   - Branded Type による URL の区別
   - TypeScript の型チェック

2. **Assert（実行時）**
   - DB 保存直前のバリデーション
   - 不正なデータを即座に検出

3. **アーキテクチャ（設計）**
   - 同期フローの一本化
   - Base64 の使用箇所を3関数に限定

---

## 違反時の対応

不変条件に違反するデータが検出された場合:

1. **開発環境**: エラーをスローして即座に停止
2. **本番環境**: エラーログを記録し、データを修復
3. **移行処理**: 既存の違反データを検出・修正
