# ゴミ箱機能 (Trash / Recovery) 仕様書

## 概要
データ保護の最終防衛ラインとして、論理削除されたリソースの管理を行います。
特に、階層構造を持つフォルダシステムにおいて、子要素の復元時に親要素の整合性をどう保つか（再帰的復元）が重要なロジックとなります。

## 対象ファイル
- `src/Pages/Trash.jsx`

## 1. データ定義と取得

### 論理削除 (Soft Delete)
データは物理的には消えず、以下のフラグで管理されます。
- `isDeleted`: `true`
- `deletedAt`: 削除日時 (`Date`)
- `deletedReason` (Optional): "recovery-deduplicated" 等の理由。

### クエリロジック (`useLiveQuery`)
- **Folders**: `db.getAllFolders()` で全件取得後、`normalizeFolder` (isSilent等の揺らぎ吸収) を経て、`isDeleted === true` のものを抽出。
- **Cards**:
  1. `db.getAllCards()` で全件取得。
  2. **削除判定**: 以下のいずれかに該当するカードを表示対象とします。
     - カード自体が `isDeleted === true`。
     - カードの `folderId` が「削除済みフォルダのIDリスト」に含まれている（親が死んでいる）。

## 2. 復元ロジック (Restore Process)

「復元」ボタン押下時、`handleRestore` 関数が実行されます。
**Offline-First原則**に従い、まず LocalDB を更新し、次に Firestore へ非同期更新をかけます。

### A. 自動的な親復元 (Recursive Parent Restore)
カードやサブフォルダを復元する際、その「親フォルダ」が削除済みであると、UI上でアクセス不能になる（迷子になる）問題を防ぐため、再帰的に親を復元します。

`restoreFolderWithParents(folderId)`:
1. 対象フォルダ情報を取得。
2. `parentFolderId` が存在する場合、その親フォルダ情報を取得。
3. 親が `isDeleted === true` ならば、先に `restoreFolderWithParents(parentId)` を再帰呼び出し。
4. 親の復元完了後、自身の復元 (`isDeleted = false`) を実行。

### B. フォルダごと復元 (Bulk Restore)
削除されたフォルダの「復元」ボタンを押した場合：
1. 上記の「親復元」ロジックでフォルダ自身を復元。
2. そのフォルダに属する**すべての削除済みカード**をスキャンし、一括で `isDeleted = false` に更新します。

## 3. 完全削除 (Purge Process)
「ごみ箱を空にする」または個別削除時。この操作は取り消せません。

1. **LocalDB (Physical Delete)**:
   - `db.purge('folders', id)` / `db.purge('cards', id)` を呼び出し、IndexedDBからレコードを抹消。
2. **Firestore (Physical Delete)**:
   - `deleteDoc` を実行。
   - 失敗時（オフライン等）は `console.warn` でログ出力（※現行実装ではSyncQueueへの明示的追加は未実装のため、次回Sync時の物理削除伝播または差異吸収に依存）。

## 4. UI仕様
- **検索**: フォルダ名・カード名（`questionText` / `title`）の部分一致検索。
- **期間フィルタ**: `deletedAt` と現在時刻の差分日数でフィルタ。
- **フォルダなし警告**: `folderId` が欠損しているカードがある場合、赤枠のアラートリストとして分離表示し、気付きを与えます。
