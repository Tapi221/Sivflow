# 運用マニュアル：トラブルシューティングとデータ復旧

## 1. 緊急事態の定義 (Red Alert)

以下のような状況を「緊急事態」と定義し、本手順に従って復旧を試みます。

1.  **無限ロード**: アプリが開かない、ロゴで止まる。
2.  **データ不整合**: 端末Aと端末Bでデータが全く違う、同期が終わらない。
3.  **データ消失**: 誤って大切なフォルダを消してしまった（かつ同期されてしまった）。

---

## 2. クライアントサイド復旧手順 (Client Recovery)

ユーザー自身が実行可能な手順です。

### Level 1: 強制同期 (Force Sync)
同期がスタックしている場合、ロックを無視して強制的に同期を試みます。
1.  設定画面 -> 「同期ステータス」を開く。
2.  「今すぐ同期」ボタンを **5回連続タップ** する（隠しコマンド）。
3.  これにより `sync_locks` が無視され、強制同期が走ります。

### Level 2: データベースのリセット (Re-fetch)
ローカルDB (IndexedDB) が破損している疑いがある場合。
1.  DevTools -> Application -> Storage -> **"Clear site data"**。
2.  または、設定 -> 「高度な設定」 -> 「データベースの再構築」。
3.  **挙動**: ローカルデータが全て消去され、次回ログイン時にFirestoreから全てのデータを再ダウンロード (Full Pull) します。
    *   *注意*: 未同期のローカル変更（オフライン中の学習ログなど）は消失します。

## 3. サーバーサイド復旧手順 (Admin Recovery)

開発者がFirebaseコンソールから行う手順です。

### ロックの解除 (Unlock)
特定のユーザーの同期が完全に止まっている場合。
*   **Firestore**: `sync_locks/{userId}` ドキュメントを削除してください。
*   これにより、クライアントは次回の起動時に新しいロックを取得できます。

### データのロールバック (Time Travel)
ユーザーが誤って全データを削除した場合。
*   **Firestore**: Google Cloud Consoleの「PITR (Point-in-time recovery)」を使用。
*   直近7日間の任意の時点の状態に復元可能です。

## 4. 既知の病理 (Known Pathologies)

### "The Orphaned Image" (孤児画像問題)
*   **症状**: カードは削除したが、画像ファイルがStorageに残っている。
*   **対処**: 現状は放置でOK（コストへの影響は軽微）。将来的には `storageCleanup` バッチを実行して掃除する。

### "The Future Date" (未来日付問題)
*   **症状**: 端末の日付設定がおかしい状態で学習し、最終復習日が「2099年」になってしまった。
*   **対処**: 当該カードの `nextReviewDate` を手動（またはスクリプト）で現在日時に修正する。
