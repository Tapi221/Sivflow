# ブラウザストレージ準拠 - 運用チェックリスト

## 目的

「壊れたとき、誰が、どこを見て、どう判断するか」を明確にするための運用者向けチェックリスト。

---

## 起動時チェック

### ✅ 1. AppInitializer の実行確認

**確認方法**: コンソールログ

**正常時のログ**:
```
[AppInit:user123] Starting initialization...
[AppInit:user123] Initialization complete
```

**異常時のログ**:
```
[AppInit:user123] Health check failed: <reason>
[AppInit:user123] Rebuilding IndexedDB...
```

**判断基準**:
- `Initialization complete` が出れば正常
- `Health check failed` が出たら再構築が実行される
- `rebuild_loop` が出たら無限ループ → 手動介入が必要

---

### ✅ 2. metadata の状態確認

**確認方法**: IndexedDB を直接確認

**手順**:
1. Chrome DevTools > Application > IndexedDB
2. `flashcard_master_<userId>` > `metadata` テーブル
3. `key: 'main'` のレコードを確認

**確認項目**:
- `storageState`: `'CLEAN'` であること
- `schemaVersion`: `11` であること
- `rebuildCount`: `0` または小さい数値であること
- `lastFullSyncAt`: 最近の日時であること

**異常パターン**:
- `storageState: 'DIRTY'` → 前回異常終了
- `rebuildCount > 3` → 無限ループ
- `schemaVersion` が不一致 → 再構築が必要

---

### ✅ 3. Snapshot の Firestore 保存確認

**確認方法**: Firestore Console

**手順**:
1. Firebase Console > Firestore Database
2. `/users/{userId}/snapshots` コレクションを確認
3. 最新の Snapshot を確認

**確認項目**:
- Snapshot が存在すること
- `metadata.schemaVersion` が正しいこと
- `metadata.generationCounter` が単調増加していること
- 最大7世代保持されていること

**異常パターン**:
- Snapshot が存在しない → 移行失敗
- `schemaVersion` が不正 → データ破損
- 世代数が異常に多い → 削除処理が動いていない

## 定期チェック（週次）

### ✅ 4. ストレージ使用量の確認

**確認方法**: Chrome DevTools

**手順**:
1. Chrome DevTools > Application > Storage
2. IndexedDB の使用量を確認

**判断基準**:
- 使用量が急激に増えていないか
- QuotaExceededError のログがないか

**対処法**:
- 履歴圧縮が動いているか確認
- 不要なデータを削除

---

### ✅ 5. 圧縮処理の実行確認

**確認方法**: コンソールログ

**正常時のログ**:
```
[Compression:user123] Compressed 1000 events into 30 daily summaries
```

**異常時のログ**:
```
[Compression:user123] Skipped (READ_ONLY mode)
[Compression:user123] Failed: <error>
```

**判断基準**:
- 圧縮が定期的に実行されているか
- 圧縮前後の件数差が妥当か
- 実行時間が異常に長くないか

### ✅ 6. READ_ONLY モードの確認

**確認方法**: コンソールログ

**異常時のログ**:
```
[Storage:user123] Entered READ_ONLY mode: <reason>
[Storage:user123] Write blocked (READ_ONLY): <context>
```

**判断基準**:
- READ_ONLY モードに入っていないか
- 入っている場合、原因は何か

**対処法**:
1. ストレージ容量を確認
2. 不要なデータを削除
3. ページをリロード

## トラブルシューティング

### 🔥 Case: アプリが起動しない

**チェック順序**:
1. コンソールで `[Auth] Fatal setup error:` を確認
2. IndexedDB が有効か確認
3. ストレージ容量を確認
4. 別ブラウザで試す

---

### 🔥 Case: データが消えた

**チェック順序**:
1. コンソールで `[AppInit] Rebuilding IndexedDB...` を確認
2. 再構築が実行されたか確認
3. Firestore にデータがあるか確認
4. Snapshot から復元

---

### 🔥 Case: 書き込みができない

**チェック順序**:
1. コンソールで `[Storage] Write blocked (READ_ONLY)` を確認
2. READ_ONLY モードに入っているか確認
3. ストレージ容量を確認
4. ページをリロード

---

### 🔥 Case: 無限再構築ループ

**チェック順序**:
1. コンソールで `rebuild_loop` を確認
2. metadata の `rebuildCount` を確認
3. クラウド同期が失敗していないか確認
4. metadata を手動でリセット
