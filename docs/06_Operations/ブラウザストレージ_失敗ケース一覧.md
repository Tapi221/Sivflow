# ブラウザストレージ準拠 - 失敗ケース一覧と対処法

## 目的

このドキュメントは「壊れたとき、誰が、どこを見て、どう判断するか」を明確にするためのものです。

---

## 想定失敗ケースと挙動

### ❌ Case 1: IndexedDB open 失敗

**発生条件**:
- ブラウザの IndexedDB が無効化されている
- プライベートモードで IndexedDB が使えない
- ストレージ容量が完全に枯渇

**挙動**:
- `initializeDB()` で例外発生
- AuthContext の try/catch でキャッチ
- `syncStatus = 'error'` に遷移

**ユーザーへの影響**:
- アプリが起動しない
- エラー画面が表示される

**対処法**:
1. コンソールで `[Auth] Fatal setup error:` を確認
2. ブラウザ設定で IndexedDB が有効か確認
3. ストレージ容量を確認
4. 別ブラウザで試す

**ログ出力例**:
```
[Auth] Fatal setup error: Error: Failed to execute 'open' on 'IDBFactory'
```

---

### ❌ Case 2: metadata テーブル破損

**発生条件**:
- IndexedDB のスキーマ不整合
- 手動での DB 操作
- ブラウザのバグ

**挙動**:
- `checkHealth()` で `healthy: false, reason: 'missing_metadata'`
- 即破棄 → 再構築

**ユーザーへの影響**:
- 起動時に再構築が実行される
- ローカルデータが消える（クラウドから復元）

**対処法**:
1. コンソールで `[AppInit] Health check failed: missing_metadata` を確認
2. 再構築が自動実行されるのを待つ
3. クラウド同期が完了していれば復元される

**ログ出力例**:
```
[AppInit:user123] Health check failed: missing_metadata
[AppInit:user123] Rebuilding IndexedDB...
```

---

### ❌ Case 3: Snapshot Firestore 側が壊れている

**発生条件**:
- Firestore のデータ破損
- アプリのバグで不正なデータを保存
- 手動での Firestore 操作

**挙動**:
- `getStoredSnapshots()` で例外発生
- または不正なデータを取得

**ユーザーへの影響**:
- Snapshot の復元ができない
- 最悪の場合、不正なデータで上書き

**対処法**:
1. Firestore Console で `/users/{userId}/snapshots` を確認
2. `metadata.schemaVersion` が正しいか確認
3. 不正なドキュメントを削除
4. 最新の正常な Snapshot を特定

**ログ出力例**:
```
[Snapshot] Failed to load snapshots: FirebaseError: ...
```

**🔥 重要**: Snapshot の version 管理と巻き戻し機能が未実装
- **TODO**: Snapshot に `version` フィールドを追加
- **TODO**: restore 前の local backup を作成
- **TODO**: 最低一世代の巻き戻し機能

### ❌ Case 4: 再構築途中でタブが落ちた

**発生条件**:
- ブラウザクラッシュ
- タブを誤って閉じた
- ネットワーク切断

**挙動**:
- `markDirty()` が実行済み
- `markClean()` が未実行
- 次回起動時に DIRTY 状態を検知 → 再度再構築

**ユーザーへの影響**:
- 次回起動時に再び再構築が実行される
- データの不整合は発生しない（DIRTY デフォルト設計）

**対処法**:
1. コンソールで `[AppInit] Health check failed: dirty_shutdown` を確認
2. 再構築が自動実行されるのを待つ

**ログ出力例**:
```
[AppInit:user123] Health check failed: dirty_shutdown
[AppInit:user123] Rebuilding IndexedDB...
```

**設計の正しさ**: DIRTY デフォルトにより、途中終了を自動検知

### ❌ Case 5: userId 切り替え中に例外発生

**発生条件**:
- ログアウト → ログイン中に例外
- 複数アカウント切り替え中にエラー

**挙動**:
- `StorageStateManager` は userId ごとに状態を保持
- 前ユーザーの状態が残る可能性

**ユーザーへの影響**:
- 状態が汚染される可能性
- READ_ONLY 状態が引き継がれる可能性

**対処法**:
1. ログアウト時に `StorageStateManager.reset(userId)` を呼ぶ
2. AuthContext のログアウト処理で状態をクリア

**🔥 重要**: 現在の実装では明示的な reset が未実装
- **TODO**: AuthContext のログアウト処理に `StorageStateManager.reset()` を追加

### ❌ Case 6: 初回ログインで Snapshot 移行だけ失敗

**発生条件**:
- Firestore への書き込み権限がない
- ネットワークエラー
- Firestore のクォータ超過

**挙動**:
- `migrateFromLocalStorage()` で例外
- try/catch でキャッチ → ログ出力のみ
- アプリは継続

**ユーザーへの影響**:
- LocalStorage の Snapshot が Firestore に移行されない
- 次回起動時に再度移行を試みる

**対処法**:
1. コンソールで `[AppInit] Snapshot migration failed:` を確認
2. Firestore の権限設定を確認
3. 手動で Snapshot を Firestore に移行

**ログ出力例**:
```
[AppInit:user123] Snapshot migration failed: FirebaseError: Missing or insufficient permissions
```

**🔥 重要**: 移行失敗時のリトライ戦略が未定義
- **TODO**: 移行失敗時のフラグを metadata に保存
- **TODO**: 次回起動時に再試行

### ❌ Case 7: 圧縮が無限ループ

**発生条件**:
- `compressByDay()` のバグ
- 圧縮データの保存先が未定義
- 削除処理が実行されない

**挙動**:
- バックグラウンドで無限に実行
- CPU 使用率が上がる
- ユーザーは気づかない

**ユーザーへの影響**:
- バッテリー消費が増える
- ブラウザが重くなる

**対処法**:
1. コンソールで `[Compression]` ログを確認
2. 実行時間が異常に長い場合は中断
3. `requestIdleCallback` のタイムアウトを設定

**🔥 重要**: 圧縮の実行時間監視が未実装
- **TODO**: 圧縮開始・終了時刻をログ
- **TODO**: 実行時間が 10 秒を超えたら警告
- **TODO**: 圧縮前後の件数差をログ

### ❌ Case 8: Firestore が一時的に落ちている

**発生条件**:
- Firestore のメンテナンス
- ネットワーク障害
- Firebase のクォータ超過

**挙動**:
- Snapshot の保存・取得が失敗
- 同期が失敗

**ユーザーへの影響**:
- Snapshot が保存されない
- 同期が停止する

**対処法**:
1. コンソールで Firestore エラーを確認
2. Firebase Status Dashboard を確認
3. ネットワーク接続を確認
4. リトライを待つ

**🔥 重要**: Firestore エラー時のリトライ戦略が未定義
- **TODO**: 指数バックオフでリトライ
- **TODO**: 最大リトライ回数を設定

### ❌ Case 9: QuotaExceededError 後の復帰

**発生条件**:
- ストレージ容量が枯渇
- READ_ONLY モードに遷移
- ユーザーがデータを削除

**挙動**:
- READ_ONLY モード中は書き込み拒否
- 容量が回復しても自動復帰しない

**ユーザーへの影響**:
- 書き込みができない
- 手動でリロードが必要

**対処法**:
1. コンソールで `[Storage] Entered READ_ONLY mode:` を確認
2. ブラウザのストレージを削除
3. ページをリロード

**🔥 重要**: READ_ONLY からの自動復帰が未実装
- **TODO**: 定期的に容量をチェック
- **TODO**: 容量が回復したら NORMAL に戻す

---

### ❌ Case 10: 無限再構築ループ

**発生条件**:
- 再構築が常に失敗する
- クラウド同期が常に失敗する
- metadata の rebuildCount が増え続ける

**挙動**:
- `rebuildCount > 3` で検知
- `checkHealth()` が `healthy: false, reason: 'rebuild_loop'` を返す

**ユーザーへの影響**:
- アプリが起動しない
- エラー画面が表示される

**対処法**:
1. コンソールで `[AppInit] Health check failed: rebuild_loop` を確認
2. metadata の rebuildCount を手動でリセット
3. 根本原因（クラウド同期失敗等）を解決

**ログ出力例**:
```
[AppInit:user123] Health check failed: rebuild_loop (count: 4)
```

**設計の正しさ**: 無限ループを検知して停止
