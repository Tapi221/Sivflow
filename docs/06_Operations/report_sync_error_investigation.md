# 同期エラーおよび画面空白化問題の調査報告書

## 1. 現状の問題点

1. **同期エラーの頻発**: ネットワークの不安定さやマージ競合により、ユーザーに不快なエラー通知が頻繁に表示される。
2. **画面の完全な空白化**: 同期エラー発生時（または特定のデータ状態時）に、フォルダ一覧やフォルダ詳細画面が完全に空白になり、操作不能になる。
3. **データアクセスの喪失**: 画面が表示されないため、ユーザーが自身の学習データにアクセスできない致命的な状況が発生している。

## 2. 原因分析

### A. 画面空白化の主原因：無限再帰によるクラッシュ
コード調査の結果、フォルダ階層の計算ロジックに**循環参照（ループ）への考慮がない**箇所が複数見つかりました。

- `src/Components/folder/FolderTree.tsx` の `getCardCount`
- `src/Pages/Folders.jsx` の `getDescendantStats`
- `src/Pages/FolderView.jsx` の `totalDescendantCardCount`

**シナリオ**:
同期プロセス中に、一時的な不整合やマージミスにより「フォルダAの親がフォルダBで、フォルダBの親がフォルダA」というループが発生すると、上記の関数が無限に自己呼び出しを行い、スタックオーバーフローを引き起こしてReactコンポーネントがクラッシュ、画面が空白になります。

### B. 同期エラー頻発の原因
- **マージ戦略の単純さ**: `DiffEngine.ts` がフィールドごとの単純な比較を行っており、複雑な不整合（親子関係の矛盾など）をバリデーションしていません。
- **リトライ不足**: ネットワークの一時的な瞬断に対する指数バックオフなどの洗練されたリトライ機構が不十分です。
- **フル同期の未実装**: `SyncServiceV2.ts` の `forceFullResync` が `// 実装は省略` となっており、致命的な不整合を自ら修復する手段がありません。

## 3. 解決策と優先度

| 優先度 | 対策内容 | 難易度 | 効果 |
| :--- | :--- | :--- | :--- |
| **最高** | **無限再帰ガードの実装**: `Set` 等を用いて巡回済みのIDを追跡し、無限ループを防ぐ。 | 低 | **極大** (画面空白化を防ぐ) |
| **高** | **Error Boundary の導入**: 万が一コンポーネントがクラッシュしても、アプリ全体ではなく部分的なエラー表示に留める。 | 中 | 高 (ユーザビリティ維持) |
| **中** | **フル同期の完全実装**: `forceFullResync` を実装し、不整合をクラウドから再取得して修復可能にする。 | 中 | 高 (データ整合性) |
| **中** | **リトライロジックの強化**: 指数バックオフ付きのリトライキューを導入する。 | 中 | 中 (エラー頻度の低下) |

## 4. 実装アプローチ

### 無限再帰ガードの例
```typescript
const getDescendantIds = (rootId, visited = new Set()) => {
  if (visited.has(rootId)) return []; // 既知のIDならループとして中断
  visited.add(rootId);
  // ... 既存の再帰処理 ...
};
```

### 改善後のアーキテクチャ案
1. **Local第一主義 (Offline First)**: 同期中であっても常にローカルキャッシュを表示し続ける。
2. **楽観的UI (Optimistic UI)**: 保存操作は即座にUIに反映し、同期はバックグラウンドで行う。
3. **データバリデーション**: 同期で取得したデータが循環参照を構成しないか検証し、異常があれば自動で修復（親をnullにする等）する。
