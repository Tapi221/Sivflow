# ブラウザストレージ分析レポート

## 概要
`docs/01_Database/ブラウザストレージ設計指針.md`（以下、設計指針）と、現在の実装（`src/services/localDB.ts` 等）を比較分析しました。

## 分析サマリー

| 項目 | 一致状況 | 詳細 |
|---|---|---|
| **基本アーキテクチャ** | ✅ 一致 | 「Source of Truth = クラウド」「IndexedDB = キャッシュ/作業領域」という基本思想は実装と合致します。`SyncService` によりクラウド主導の同期が実現されています。 |
| **データ保存ポリシー** | ✅ 一致 | 画像の Blob URL 永続化防止ロジック (`normalizeUploadedImages`) や、LocalStorage の限定利用などのポリシーは遵守されています。 |
| **メタデータ管理** | ⚠️ 部分的一致 | 最新スキーマ (v11) で `metadata` テーブルが追加されていますが、設計指針にある「厳密な整合性チェック（バージョン不整合時の即自爆など）」はコードレベルでは実装されていません。むしろ `importFromDatabase` などの「データ救出・復旧」ロジックが充実しています。 |
| **容量エラーハンドリング** | ⚠️ 未実装 | 設計指針にある「QuotaExceededError 発生時に Read-Only モードへ遷移する」といった体系的なエラーハンドリング機構は `LocalDB` クラス内に見当たりません。 |
| **競合制御 (Phase 1)** | ℹ️ 実装先行 | Phase 1 で実装された高度な競合制御テーブル（`syncQueue`, `conflicts` 等）に関する記述が設計指針にはまだありません。 |

## 詳細分析

### 1. 「捨てる」vs「救う」の設計思想
設計指針では**「迷ったら捨てる」**（不整合があれば即破棄して再同期）という厳格なポリシーが掲げられていますが、実際のコード(`DataRescuePanel`や `importFromDatabase`)は**「なんとかして救う」**（泥臭い復旧処理）方向に進化しています。
これはユーザー体験（「せっかく作ったデータが消える」ことへの恐怖）を現実的に考慮した結果であると推測されますが、ドキュメントとコードの思想に乖離があります。

### 2. スキーマバージョン管理
`LocalDB` のスキーマ定義はバージョン11まで適切に管理されており、同期に必要なインデックス（`[userId+updatedAt]` 等）も整備されています。

### 3. 未実装の高度な機能
以下の機能は設計指針に記述されていますが、現状のコードでは確認できませんでした：
- **厳密なメタデータ運用**: `expectedEntityCounts` と実数の照合ロジックなど。
- **階層別キャッシュ管理**: 検索インデックスなどを「高コスト生成レイヤー」として分離し、容量不足時に優先削除するロジック。
- **QuotaExceededError のグローバルハンドリング**: エラー発生時にアプリ全体を縮退運転させるステート管理。

## 推奨アクション

### ドキュメントの更新
設計指針を現在の「リカバリ重視」の実装に合わせて更新し、Phase 1の実装内容（競合制御テーブル等）を反映させることを推奨します。具体的には：

1. **競合制御テーブルの追記**: `syncQueue`, `conflicts`, `syncErrors`, `syncHistory` の役割定義。
2. **「即破棄」ポリシーの緩和**: 「可能な限り復旧を試み、最終手段として再同期する」という現状のアプローチに表現を修正。
3. **未実装項目の明記**: 容量制限ハンドリングなどは「将来のロードマップ」として区別。

### コードの改善（将来的な課題）
- **容量エラー検知**: `QuotaExceededError` をキャッチする統一的な仕組みの導入（まずはログ出力から）。
- **メタデータ活用の開始**: 追加された `metadata` テーブルを活用し、最終同期時刻以外の健全性指標（レコード数など）を記録開始する。
