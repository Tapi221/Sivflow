# ローカルDBアーキテクチャ監査レポート

**監査日**: 2026-02-01
**対象**: Dexie.js (IndexedDB) / localStorage / Sync System
**監査レベル**: **Critical (運用継続性リスクあり)**

---

## 1. 監査概要 (Audit Summary)

本システムは Dexie.js を採用した高度なオフラインファースト設計を有するが、**「障害発生時の運用プロセス」および「データの最終的な正」を定義する意思決定フローが欠落している。**
特に、localStorage への安易な依存と、Queue 肥大化時の**「回復不能な論理破損」**リスクは、サービス運用において時限爆弾となり得る。

本監査が **Critical** と判断された理由は、**「単一障害によりユーザー判断を介さずデータ消失に至る経路が存在する」**ためである。

### 最終評価

- **技術選定**: A (適切)
- **設計思想**: A- (基本構造は堅牢)
- **運用・事故対応**: B- (手順未定義だったが定義により向上)
- **リスク管理**: **Critical** (単一障害点あり)

## 0. Data Authority Matrix (データ正本定義)

本システムにおける「信頼できる唯一の情報源 (Single Source of Truth)」を以下の通り定義する。これはいかなる実装・運用においても最優先されるルールである。

| 状態 | 正本 (Authority) | 備考 |
| :--- | :--- | :--- |
| **Online (通常)** | **Server (Firestore)** | ローカルはあくまでキャッシュ。同期成功が正。 |
| **Offline** | **LocalDB + ClientSeq** | オフライン操作の生データと順序が正本。 |
| **Recovery** | **User Choice** | 競合・破損時はユーザーの明示的な選択が正本。勝手な自動マージは禁止。 |
| **Public Data** | **Server** | 共有データの整合性はサーバー側ルールが絶対。 |

## 2. 重要リスク評価：回復不能な論理破損シナリオ

### 2.1 Dexie Schema 破損 / Upgrade 失敗

- **発生シナリオ**: データベースファイル破損、またはマイグレーションロジックのバグ。
- **意思決定フロー (復元プロトコル)**:
  1. **ユーザー明示確認**: 「データ破損を検知しました。サーバーから復元しますか？」ダイアログを表示。
  2. **Survival First**: 完全なフルダンプは諦め、**生存率を優先**する。`syncQueue`, `conflicts`, `syncMetadata` の重要3テーブルのみを抽出し、JSONとしてクリップボードコピーまたはDLさせる。「完全ではありません」と正直に伝える。
  3. **Wipe & Restore**: ローカルデータを完全破棄し、クラウド正本から再同期する。

### 2.2 Operation Queue Explosion (論理破損)

- **発生シナリオ**: オフラインで数万件の操作を行い、再同期時にタイムアウトや部分的な失敗が発生する。
- **対策**:
  - localStorage の使用を即時停止（容量制限による保存失敗の回避）。
  - **Configurable Limits**: 閾値はハードコードせず、`syncMetadata` 内の `queueSoftLimit` (Default: 10,000) および `queueHardLimit` で定義し、動的に変更可能とする。
  - **Honest UX**: 「99,800件」のような進捗に加え、**「処理には数十分かかる可能性があります。このままアプリを閉じても、次回ここから再開します」**という安心感のあるメッセージを表示する。

### 2.3 Conflicts の滞留と自動解決ポリシー

- **現状の問題**: 「30日で強制解決」という根拠のない一律ルールは採用できない。
- **データ種別ポリシー**:

| 種別 | ポリシー |
| :--- | :--- |
| **学習ログ・統計** | **Local Wins** |
| **共有・公開データ** | **Server Wins** |
| **カード編集・作成** | **Manual / Notify** |

## 3. localStorage の存在意義と制限 (Allowed List)

**定義**:

> **localStorage は「DBが死んでもUIを止めないための最終避難所」であり、原則利用は禁止する。**

**Allowed List** (これ以外は原則禁止):

- `emergencyModeFlag`: 重大エラー発生時のセーフモード起動フラグ
- `lastSafeBootTimestamp`: 起動ループ検知用
- `uiPreferences`: 256Byte以下のUI設定（テーマ、言語など）
- `auth_token`: (Firebase SDK管理のものに限る)

## 4. テーブル棚卸しと責務定義

| テーブル名 | 1行責務定義 | 現状の評価・改善 |
| :--- | :--- | :--- |
| `syncHistory` | **人が読むための同期履歴ログ。** | 適切。 |
| `syncErrors` | **再試行ロジックが参照するエラー状態管理。** | リトライ完了時に即削除すること。 |
| `syncMetadata` | **同期プロトコル制御のための機械用メタデータ。** | 最重要。`clientSeq`や上限値設定もここに持たせる |
| `syncQueue` | **オフライン時の変更操作バッファ。** | **localStorage からの即時移行対象 (Priority: High)。** |
| `conflicts` | **ユーザー介入が必要なデータの一時保管場所。** | ポリシー別自動解決の対象外のみを保持する。 |

## 5. 推奨アクションプラン

### Phase 1: 止血 (Emergency Fix)

1. **Operation Queue の IndexedDB 移行**: 論理破損の温床である localStorage queue を撤廃。
2. **localStorage 禁止ルールの適用**: コードベースから 1KB 以上の書き込み箇所を特定し排除。

### Phase 2: 安全化 (Resilience)

1. **Emergency Restore UI の実装**: 生存率優先のエクスポート機能。
2. **トランザクション実装**: データ更新とキュー追加のアトミック化。

### Phase 3: 運用整備 (Operations)

1. **ランブックの作成**: 下記「事故シナリオ別ランブック」を運用ドキュメント化する。

## 6. [追加] 事故シナリオ別ランブック (Runbook)

コード修正ではなく、**「障害発生時に人間がどう動くか」**の手順書。
※ UI表現の詳細は別途 UX 仕様書に委ねる。本書は「行動判断」を定義する。

### 6.1 DB Open 失敗 / スキーマ破損時

1. **検知**: アプリ起動時の `db.open()` catch ブロックで致命的エラー画面を表示。
2. **一次対応**: 画面上の「緊急データ退避」ボタンを押させ、重要テーブルのみを軽量エクスポートさせる。
3. **復旧**: 「クラウドから復元」を実行。ローカルDBを削除(`Dexie.delete()`)し、再同期を行う。

### 6.2 Sync Queue 閾値超過 (Queue Explosion)

1. **検知**: 同期開始前の件数チェックで `queueSoftLimit` 超過を検知。
2. **制限発動**: 通常同期を停止し、**「非同期バックグラウンドモード」**に移行。
3. **UX**: ユーザーには「完了予測時間」と「中断しても安全であること」を伝え、操作をブロックしない。

### 6.3 Firestore (Remote) 障害時

1. **検知**: 連続する 5xx エラーまたはネットワークタイムアウト。
2. **挙動**: **「完全オフラインモード」**へ自動移行.
3. **復帰**: 1時間ごとのポーリング（またはユーザーの手動リトライ）でのみ復帰を試行。
