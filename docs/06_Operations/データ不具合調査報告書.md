# データ表示不具合およびフォルダ作成エラーの調査報告書

## 1. 概要
ユーザーから報告された「新規フォルダ作成が反映されない」「以前作成したカードが表示されない」という不具合について、ソースコードレベルでの調査を実施しました。複数の重大なバグおよび設計上の懸念事項が特定されました。

## 2. 調査結果

### A. 同期ロジックの致命的欠陥 (CloudSyncAdapter.ts)
Firestore からのデータ取得 (`pullDiff`) において、時刻の比較に失敗している可能性が高いことが判明しました。
- **原因**: Firestore 側で `serverTimestamp()` (Timestampオブジェクト) として保存されている `updatedAt` フィールドに対し、プログラム側から `number` (ミリ秒) で比較クエリを投げています。Firestore の型感度により、この比較は期待通りに動作せず、データが 0 件として返される原因となります。
- **影響**: 他のデバイスで作成したデータや、再ログイン後のデータがローカルに一切降ってこない「表示消失」の原因となります。

### B. 初期化プロセスの設計不備とデータ消失リスク (AppInitializer.ts / IndexedDBRebuildOrchestrator.ts)
IndexedDB の健全性チェックに失敗した際の「再構築」処理が未実装 (TODO) であることが判明しました。
- **原因**: `AppInitializer` は DB が「不健全 (DIRTY)」であると判断すると、即座に `db.delete()` を実行してデータを破棄します。しかし、その後の `IndexedDBRebuildOrchestrator.rebuild` は中身が TODO のままであり、クラウドからのデータ復旧を行いません。
- **影響**: アプリが異常終了したり、同期途中で閉じられた後に再起動すると、ローカルデータがすべて消去され、空の状態で起動してしまいます。

### C. 同期オーケストレーションの不完全さ (SyncServiceV2.ts)
`AuthContext` から呼び出される `performFullSync()` や `synchronize()` が、実際には「クラウドからの取得 (Pull)」をトリガーしていません。
- **原因**: `performFullSync` が `sync()` メソッドを呼び出していますが、この `sync()` はローカルの送信待ちキュー (Push) の処理しか行っておらず、`pullDiff` を呼び出していません。
- **影響**: 手動同期ボタンを押しても、クラウド上の新しいデータが降ってきません。現在は `App.tsx` での `performStartupSync()` のみに Pull 処理が依存しています。

### D. フォルダ作成・カード表示の整合性 (LocalDB 修復ロジック)
`localDB.ts` の `repairDataIntegrity` が非常に攻撃的な復旧を行っている可能性があります。
- **内容**: 全てのカードを強制的に一つの「正規フォルダ」にまとめようとする挙動があり、意図しないフォルダ移動が発生している可能性があります。
- **不具合との関連**: 作成したはずのフォルダではなく、別の復旧フォルダにカードが入ってしまった場合、ユーザーからは「消えた」ように見える可能性があります。

## 3. 推奨される修正アクション
1. **`CloudSyncAdapter.ts`**: 時刻比較を `Timestamp.fromMillis()` を使用するように修正。
2. **`IndexedDBRebuildOrchestrator.ts`**: TODO を解消し、Firestore からの全データ取得ロジックを実装。
3. **`SyncServiceV2.ts`**: `sync()` メソッド内でも `pullDiff` を実行するようにオーケストレーションを改善。
4. **`AuthContext.tsx`**: 初期化とデータ取得のレースコンディションを防止するため、UI描画前のデータ存在をより厳密に保証。

## 4. 結論
現在、データは Firestore 上には残っている可能性が高いですが、アプリ側の同期エンジンの不備により、ローカルへの「ダウンロード」が正常に行われない状態です。上記の修正を行うことで、データの可視性が回復すると考えられます。
