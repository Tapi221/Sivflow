# 開発ガイドライン

Flash Master の開発におけるコーディング規約、ベストプラクティス、命名規則をまとめたドキュメントです。

## コーディング規約

### TypeScript

#### 型定義

- インターフェース名は PascalCase
- 型エイリアスは PascalCase
- 型ファイルは `types/` ディレクトリに配置

```typescript
// ✅ Good
interface UserSettings {
  theme: 'light' | 'dark';
}

type UserId = string;

// ❌ Bad
interface userSettings {}
type user_id = string;
```

#### 変数・関数名

- 変数・関数名は camelCase
- 定数は UPPER_SNAKE_CASE（ただし、React コンポーネント内の定数は camelCase）

```typescript
// ✅ Good
const userName = 'John';
function getUserData() {}
const MAX_RETRY_COUNT = 3;

// ❌ Bad
const user_name = 'John';
function GetUserData() {}
```

#### コンポーネント名

- React コンポーネントは PascalCase
- コンポーネントファイル名も PascalCase（例: `UserSettings.tsx`）

```typescript
// ✅ Good
function UserSettings() {}
export default UserSettings;

// ❌ Bad
function userSettings() {}
```

### React

#### コンポーネント定義

- 関数コンポーネントを使用
- `export default` を使用（1ファイル1コンポーネント推奨）

```typescript
// ✅ Good
function CardList() {
  return <div>...</div>;
}

export default CardList;

// ❌ Bad
const CardList = () => <div>...</div>;
```

#### Hooks

- カスタムフックは `use` で始める
- Hooks はコンポーネントのトップレベルで呼び出す

```typescript
// ✅ Good
function useCardData(cardId: string) {
  const [data, setData] = useState(null);
  // ...
  return data;
}

function CardDetail({ cardId }: { cardId: string }) {
  const cardData = useCardData(cardId);
  // ...
}

// ❌ Bad
function getCardData(cardId: string) {} // use がついていない
```

#### 状態管理

- ローカル状態: `useState`
- 複数コンポーネント間の状態: Context API または状態管理ライブラリ
- サーバー状態: 必要に応じて React Query 等を検討

### ファイル構造

#### ディレクトリ構成

```
src/
├── components/          # 再利用可能なコンポーネント
│   ├── layout/         # レイアウトコンポーネント
│   ├── ui/             # UI コンポーネント（ボタン、入力欄等）
│   └── forms/          # フォームコンポーネント
├── pages/              # ページコンポーネント
├── hooks/              # カスタムフック
├── contexts/           # React Context
├── services/           # API 通信、Firebase 操作
├── utils/              # ユーティリティ関数
├── types/              # TypeScript 型定義
└── constants/          # 定数
```

#### ファイル命名

- コンポーネント: `PascalCase.tsx`（例: `CardList.tsx`）
- ユーティリティ: `camelCase.ts`（例: `formatDate.ts`）
- 型定義: `camelCase.ts`（例: `userTypes.ts`）
- 定数: `UPPER_SNAKE_CASE.ts` または `camelCase.ts`（例: `apiConstants.ts`）

### スタイリング（TailwindCSS）

#### クラス名の順序

1. レイアウト（flex, grid, block）
2. サイズ（w-, h-）
3. スペーシング（p-, m-）
4. タイポグラフィ（text-, font-）
5. 色（bg-, text-, border-）
6. その他（rounded, shadow）

```tsx
// ✅ Good
<div className="flex items-center gap-4 p-4 text-lg bg-blue-500 rounded-lg">

// ❌ Bad（順序が不規則）
<div className="bg-blue-500 flex p-4 rounded-lg text-lg">
```

#### 再利用可能なスタイル

- 共通スタイルはコンポーネント化
- 繰り返し使用するパターンはユーティリティクラスに抽出

```tsx
// ✅ Good
function Button({ children, variant }: ButtonProps) {
  const baseClasses = 'px-4 py-2 rounded';
  const variantClasses = {
    primary: 'bg-blue-500 text-white',
    secondary: 'bg-gray-200 text-gray-800',
  };
  return <button className={`${baseClasses} ${variantClasses[variant]}`}>{children}</button>;
}
```

### Firebase 操作

#### Firestore 読み取り

- 必要なデータのみを取得
- リアルタイムリスナーは適切にクリーンアップ

```typescript
// ✅ Good
useEffect(() => {
  const unsub = onSnapshot(doc(db, 'userStats', userId), (doc) => {
    setStats(doc.data());
  });
  return () => unsub();
}, [userId]);

// ❌ Bad（クリーンアップなし）
useEffect(() => {
  onSnapshot(doc(db, 'userStats', userId), (doc) => {
    setStats(doc.data());
  });
}, [userId]);
```

#### エラーハンドリング

- すべての Firebase 操作でエラーハンドリングを実装
- ユーザーに分かりやすいエラーメッセージを表示

```typescript
// ✅ Good
try {
  await updateDoc(doc(db, 'userSettings', userId), { theme: 'dark' });
  toast.success('設定を更新しました');
} catch (error) {
  console.error('設定更新エラー:', error);
  toast.error('設定の更新に失敗しました');
}
```

### コメント

#### コードコメント

- 複雑なロジックにはコメントを追加
- 「何をしているか」ではなく「なぜそうしているか」を説明
- TODO コメントは期限を明記

```typescript
// ✅ Good
// 忘却曲線に基づいて次回復習日を計算
// Lv.0: 1日、Lv.1: 3日、Lv.2: 7日...
const nextReviewDate = calculateNextReviewDate(currentLevel, lastStudyDate);

// TODO: 2026-02-01 までに最適化が必要
const slowFunction = () => {};

// ❌ Bad（自明なコメント）
// 変数 x に 1 を代入
const x = 1;
```

## Git 運用

### ブランチ戦略

- `main`: 本番環境用（保護ブランチ）
- `develop`: 開発環境用
- `feature/機能名`: 機能開発用
- `fix/バグ名`: バグ修正用
- `hotfix/緊急修正名`: 緊急修正用

### コミットメッセージ

- 日本語または英語で記述
- プレフィックスを使用

```
feat: カード作成機能を追加
fix: ログインエラーを修正
refactor: 統計集計ロジックをリファクタリング
docs: README を更新
style: コードフォーマットを修正
test: ユニットテストを追加
chore: 依存関係を更新
```

### プルリクエスト

- 1つの PR は1つの機能・修正に絞る
- 説明欄に変更内容、テスト方法を記載
- レビューを依頼する前に自己レビュー

## パフォーマンス

### 最適化のポイント

1. **コード分割**: React.lazy でページ単位のコード分割
2. **メモ化**: useMemo, useCallback を適切に使用
3. **画像最適化**: 遅延読み込み、適切なサイズ
4. **Firestore read 最小化**: 事前集計済みデータを使用

### 避けるべきパターン

```typescript
// ❌ Bad（不要な再レンダリング）
function Component({ items }: { items: Item[] }) {
  const sorted = items.sort((a, b) => a.name.localeCompare(b.name));
  return <List items={sorted} />;
}

// ✅ Good
function Component({ items }: { items: Item[] }) {
  const sorted = useMemo(
    () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
    [items]
  );
  return <List items={sorted} />;
}
```

## セキュリティ

### データアクセス

- 常に `userId` でデータをフィルタリング
- Firestore セキュリティルールを確認
- クライアントサイドでの権限チェックは補助的なもの（サーバーサイドルールが優先）

### 機密情報

- 環境変数に API キーを保存（`.env.local`）
- `.env.local` を Git にコミットしない
- 本番環境の環境変数は Firebase Hosting で管理

## ローカルファースト・アーキテクチャ (v2.1)

FlashcardMaster は、ネットワークが不安定な環境でも快適に使用できるよう、ローカルDB（IndexedDB）を主軸とした設計を行っています。

### 基本原則

1.  **Local Source of Truth**: UIから参照・操作するデータは常に `LocalDB` です。Firestore への直接アクセスは、同期プロセス以外では原則禁止です。
2.  **Explicit Sync Triggers**: `LocalDB` の `addItem`, `updateItem`, `softDelete` 系のメソッドを使用してください。これらは実行時に自動的に `syncQueue` にタスクを積み、クラウド同期をトリガーします。
3.  **No Direct Cloud Write**: 開発効率のために UI コンポーネントから直接 Firestore の `doc()` や `updateDoc()` を呼ぶことは避けてください。同期キューを介さない書き込みは、デバイス間の整合性を破壊します。

### カード・フォルダ操作の推奨パターン

```typescript
// ✅ Good: LocalDB 経由で操作（自動同期される）
const db = new LocalDB(userId);
await db.addItem('cards', newCard);

// ❌ Bad: Firestore を直接操作
await addDoc(collection(firestore, 'cards'), newCard);
```
