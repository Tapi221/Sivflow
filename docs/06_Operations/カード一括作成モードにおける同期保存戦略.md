# カード一括作成モードにおける同期・保存戦略

## 1. 基本設計：Local-First（ローカルファースト）
本アプリのカード作成（一括作成モードを含む）は「ローカルファースト」の原則に基づいています。ユーザーの入力は即座にブラウザのローカルデータベースに保存され、ネットワークの状態に関わらず遅延のない操作感を提供します。

## 2. 四段階の永続化・同期レイヤー

### レイヤー1：エディタ内ドラフト (localStorage)
- **対象**: 各カードエディタの入力内容。
- **機能**: 入力中のクラッシュ保護。同期対象外。

### レイヤー2：セッション・スタック (sessionStorage)
- **対象**: 一括作成モードのエディタのリストと状態。
- **機能**: ページリロード時の中断復帰。同期対象外。

### レイヤー3：ローカルDB (IndexedDB / Dexie) ★同期の起点★
- **対象**: 保存済みの確定データ。
- **機能**: **唯一の信頼できるデータソース**。更新時に自動的に `syncQueue` へタスクを積みます。

### レイヤー4：クラウド同期 (Firestore / V2)
- **仕組み**: `QueueManager`, `DiffEngine`, `CloudSyncAdapter` を使用した堅牢な同期。
- **順序**: 競合を防ぐため「受信（Pull）ファースト」で動作します。

## 3. 同期フローの詳細 (V2)

### 起動時同期 (`performStartupSync`)
1. **Pull & Apply**: 前回同期以降のクラウド上の変更を `DiffEngine` でマージしてローカルに適用します。
2. **Push**: ローカルで発生した未同期の変更をクラウドへプッシュします。
3. **Checkpoint**: 成功後に最終同期時刻を更新します。

### バックグラウンド同期
- IndexedDB の更新（作成・変更・削除）を検知し、非同期で `SyncServiceV2` をトリガーします。
- UIスレッドをブロックせず、バックグラウンドでのバッチ送信を行います。

## 4. 特筆すべき挙動・安全設計
- **データ表示不具合の解消**: 起動時にクラウドから最新データを Pull するため、新規ログイン時やデータ消失時も自動的に復元されます。
- **競合（Conflict）解決**: Last Write Wins (サーバー優先) を基本とし、競合発生時は整合性を維持しながらマージします。
- **負荷制御**: 低速ネットワーク時は同期を抑制し、キューに蓄積することで通信オーバーヘッドを最小化します。
