# カード保存不可不具合の調査報告書および修正計画

## 1. 概要

ユーザーより「カードが保存されない」という不具合が報告されました。
直近の Sync V2 実装における変更箇所を中心に調査した結果、保存処理を停止または破損させる可能性のある複数の不具合を特定しました。

## 2. 特定された問題点

### 問題 1: `SyncServiceV2.ts` における `nanoid` のインポート漏れ

- **詳細**: `SyncServiceV2.ts` 内の `applyRemoteChanges` メソッドで `nanoid()` を使用していますが、ファイル冒頭で `import { nanoid } from 'nanoid';` が不足しています。
- **影響**: 起動時同期（Startup Sync）やダウンロード処理が発生した際に `ReferenceError` でクラッシュし、アプリの初期化や同期に影響を与えます。

### 問題 2: `LocalDB.ts` の `updateItem` による画像データの消失 (重大)

- **詳細**: `updateItem` 内で `denormalizeCardForStorage` を呼び出していますが、この関数は「完全なカードオブジェクト」を想定した設計になっており、入力（changes）に画像配列が含まれない場合、空の配列 `[]` で上書きしてしまいます。
- **影響**: カードの一部（タイトルなど）を更新するたびに、既存の画像データが IndexedDB から消失します。これが「保存されない（＝データが消える）」と認識されている可能性があります。

### 問題 3: 画像不変条件チェック（Base64 禁止）による保存ブロック

- **詳細**: `LocalDB.ts` の `addItem` に、保存前の画像不変条件チェック (`assertImageArrayInvariant`) を追加しました。これは Base64 文字列の保存を厳格に禁止します。
- **影響**: カードを「保存」する際、画像がまだ Blob URL や Storage URL に変換されていない（Base64 のままの）状態だと、バリデーションエラーをスローして IndexedDB への書き込み自体を中断します。

### 問題 4: `enqueueSync` の `targetId` 取得の脆弱性

- **詳細**: `addItem` で `targetId: payload.id` を取得していますが、自動採番や ID 生成のタイミングによっては `undefined` になるリスクがあります。

---

## 3. 修正計画

### [A] `SyncServiceV2.ts` の修正

- [x] `nanoid` のインポートを追加。
- [x] `SyncContextSource` の型不整合（'startup' -> 'system'）を修正。

### [B] `LocalDB.ts` の修正

- [x] `denormalizeCardForStorage` を、Partial な更新にも対応できるよう修正（フィールドが存在する場合のみ変換）。
- [x] `updateItem` において、`denormalize` 前に既存の画像を保持するか、変換ロジックを安全にする。
- [x] `enqueueSync` において、`id` (主キー) を確実に取得するロジックへ変更。

### [C] バリデーションの緩和（移行期措置）

- [x] `assertImageArrayInvariant` でエラーを投げるのではなく、一時的に警告（`warnImageInvariant`）に留めるか、保存前に確実にサニタイズする処理を強化。

---

## 4. 検証プラン

1. **新規カード作成**: 画像を含むカードが正常に保存され、IndexedDB に書き込まれるか確認。
2. **既存カード更新**: タイトルのみ変更した際、画像が消えずに維持されるか確認。
3. **同期キュー確認**: 保存後に正常に `syncQueue` にタスクが積まれるか確認。
4. **起動時動作**: 起動時の `performStartupSync` がエラーなく完了するか確認。
