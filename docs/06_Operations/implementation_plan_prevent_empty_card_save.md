# 空カードの保存防止および自動削除の実装計画

## 概要
現在、`createCard` では空のカード作成を防ぐバリデーションが存在しますが、既存カードを更新する `updateCard` には同様のチェックがありません。
ユーザーの要望「タイトルも入力していないかつ問題側にも解答側にも入力がない(ブロックを追加していても入力なしも含む)、画像や音声も追加していないカードは存在できないようにしてください。保存されたとしてもそれはデータが消えるように」を満たすため、カード更新時に内容が空になった場合、そのカードを自動的に削除するロジックを実装します。

## 変更内容

### `src/hooks/useCards.ts`

1.  **空判定ロジックの共通化**:
    - `createCard` 内にあるバリデーションロジック（`hasBlocksContent`, `isCompletelyEmpty`）を、フック内または独立した関数として抽出・共通化します。
    - 判定基準:
        - タイトルなし
        - タグなし
        - 問題/解答のブロックに実質的なコンテンツ（テキスト、画像、音声、数式、リンク）が含まれていない
        - (Legacy) `questionText`, `answerText` が空

2.  **`updateCard` の改修**:
    - 現在の仕様（単なる更新）から、以下のフローに変更します。
        1. DBから現在のカードデータを取得（`merged` データを作成するため）。
        2. 更新データ（`Partial<Card>`）とマージして、更新後のカード状態をシミュレーション。
        3. 更新後の状態が「完全に空」であるかチェック。
        4. **空の場合**: `db.updateItem` ではなく `deleteCard` (ソフトデリート) を呼び出し、処理を終了。
        5. **空でない場合**: 通常通り `db.updateItem` を実行。

### `src/Components/card/CardEditor.tsx` (検証時の確認)

- 現在 UI 上で `isCardEmpty()` により保存ボタンが非活性化される仕様になっています。
- もし「空にして保存ボタンを押すと削除される」挙動をユーザーが期待している場合は、ボタンの `disabled` 条件を緩和する必要がありますが、今回の要望は「保存されたとしてもデータが消えるように（＝不正データを作らせない）」というガードの意味合いが強いと解釈し、まずは Backend (Hook) 側のガードを優先します。
- これにより、何らかの理由で空データが送信された際も、ゴミデータとして残らず削除されるようになります。

## 検証計画

### 自動テスト（ユニットテスト的な検証）
- 今回は `useCards` フックの修正が主となるため、ブラウザ上での動作確認を中心に行います。

### 手動検証手順
1.  **新規作成時のガード確認**:
    - 「新規作成」でタイトル等を入れずに保存ボタンが押せない（または保存しても作成されない）ことを確認（既存仕様）。
2.  **更新時の削除確認**:
    - 既存のカードを開く。
    - タイトル、テキスト、すべてのブロックを削除/空にする。
    - （UIで保存ボタンが押せない場合、一時的に `CardEditor` の制限を解除するか、コンソールから `updateCard` を叩いて検証する）
    - *補足*: `CardEditor` で空入力を許可しないと「保存」アクション自体が発火しないため、「空にして保存＝削除」を実現するには `CardEditor` 側の `disabled` 条件も外す必要があるかもしれません。ユーザーの意図が「空のカードが存在できないようにする（Backend Guard）」なのか「空にして保存したら消してほしい（UI UX）」なのかによって対応が変わります。
    - 文脈「存在できないようにしてください。保存されたとしても…」から、Backend Guard が必須です。
    - 念のため、検証用に一時的にUI制限を外して「空更新 -> 削除」が機能することを確認します。

## 懸念点・備考

    - `toggleFlag` (フラグ切替のみなので空にはならないはず)
