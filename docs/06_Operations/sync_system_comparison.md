# 同期システム比較・実装ステータス報告

## 1. 概要
本プロジェクトでは、同期システムの信頼性と拡張性を向上させるため、レガシーな `SyncService` から、オーケストレーターパターンを採用した `SyncServiceV2` への移行を進めています。
現在の調査により、V2への切り替え後にデータが表示されなくなった原因は、V2側の「クラウドからのデータ取得（ダウンロード・マージ）」が未実装であったためと判明しました。

## 2. システム構成比較

| 項目 | レガシー (SyncService) | 新システム (SyncServiceV2) |
| :--- | :--- | :--- |
| **設計思想** | モノリシック（単一クラス） | マイクロモジュール（責務分散） |
| **主なコンポーネント** | `SyncService` クラスに集約 | `QueueManager`, `DiffEngine`, `CloudSyncAdapter` 等 |
| **同期方式** | 全体同期・単純な差分同期 | バッチ処理・コンテキストを考慮した同期 |
| **ネットワーク監視** | 単純なオンライン/オフライン判定 | `NetworkMonitor` による回線品質の動的判定 |
| **セキュリティ** | 基本的なデバイスID管理 | `SecurityMonitor` による論理削除・監査ログ |
| **ログ・追跡** | ローカルDBへの履歴記録 | `TelemetryService` による詳細なトランザクションログ |

## 3. 実装ステータス詳細

### レガシー (SyncService)
- **ステータス**: **[完了]** 本番稼働実績あり。
- **実装済み機能**:
    - アップロード（双方向）
    - ダウンロード・自動マージ（プル）
    - 競合検知と解決ロジック
    - 分散ロック（二重実行防止）
    - 画像同期のオーケストレーション

### 新システム (SyncServiceV2 / Phase 2)
- **ステータス**: **[開発中 / 一部未実装]**
- **実装済み機能**:
    - 構造設計・インターフェース定義
    - `QueueManager` (タスクの永続化と優先度管理)
    - `NetworkMonitor` (回線品質に応じたバッチ制限)
    - `CloudSyncAdapter` (Firestore I/Oの抽象化)
    - アップロード処理（バッチ送信）
- **未実装・TODO項目**:
    - **ダウンロード適用ロジック**: クラウドから取得した差分を IndexedDB に反映する処理が未実装。
    - **マージロジックの統合**: `DiffEngine` は存在するが、ダウンロードフローへの組み込みが未完了。
    - **再構築オーケストレーター**: `IndexedDBRebuildOrchestrator` による初期同期フローが未実装。

## 4. 結論と今後の対応
現在、`SyncServiceV2` は「送信（アップロード）」には対応していますが、「受信（ダウンロード）」が未完成の状態です。そのため、新規ログイン時やキャッシュクリア時に Firestore からデータが降ってこない問題が発生しています。

**復旧対応:**
- 機能を復旧させるため、フラグ（`USE_SYNC_V2: false`）によりレガシーシステムへ一時的に切り戻します。

**今後の再開計画:**
- `SyncServiceV2` にレガシーの `pullChanges` 相当のロジックを移植。
- `IndexedDBRebuildOrchestrator` の実装を完了させ、初回データ取得を保証。
- 統合テストを経て、再度段階的ロールアウトを実施。
